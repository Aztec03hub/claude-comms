## 5. Step 3.8 design

### Where IntersectionObserver attaches

`ChatView.svelte` already had an IntersectionObserver for ReadReceipt `markSeen`. Wave C extended the same observer to also schedule a dwell-timer-gated `markMessageViewed(channelId, messageId)` call. Single observer, single observe-loop, dual handlers. Observer threshold stays at `0.5` (existing) — bubble must be 50%+ visible to trigger intersection.

### Dwell-time

Pinned at **1000ms** per Step 3.8 spec ("≥ 1 second visible"). Implemented via `setTimeout` keyed by `messageId` in a `Map`. On un-intersection before timer fires, the pending timer is cleared so a rapid scroll past does NOT count as a read.

### Edge cases addressed

| Case | Handling |
|---|---|
| Rapid scroll past | `entry.isIntersecting === false` branch cancels the pending `setTimeout` for that messageId |
| Channel switch mid-dwell | `channelId` is snapshotted at observer-creation time via a `$derived` on `store.activeChannel`; the `$effect` re-runs on channel change, disposing the old observer + timers and creating a new one bound to the new channel. In-flight timers fire against the OLD channel id (correct: the user viewed those bubbles in the OLD channel's viewport) |
| Re-observing same message after flicker | `dwellTimers.has(msgId)` check + `clearTimeout` before scheduling a fresh timer |
| Cleanup on unmount | `$effect` return closure disconnects observer + clears every pending timer + clears the dwellTimers Map |
| Idempotent markMessageViewed | Per-channel Set membership check; double-call is a no-op |
| Self-authored messages | Excluded from `unreadRemaining` count so an author who immediately scrolls back doesn't get stuck with unread on their own messages |
| System messages | Excluded similarly (system events aren't "unread" semantically) |

### Why NOT auto-clear on switch anymore

Per spec: switching shouldn't equal reading. Slack/Discord/Teams all require the bubble to be on-screen for a beat. The legacy auto-clear was a UX shortcut that masked real "I haven't read this yet" signal. Removing the clear means the unread badge stays until the user actually views each message bubble.

## 7. New-owner picker UX decision

Chose **native `<select>` dropdown** over custom typeahead / combobox for two reasons:

1. Participant counts per channel are typically small (< 50); a search field adds no value.
2. `<select>` carries full a11y semantics (keyboard nav, ARIA listbox role, screen-reader pronunciation) without bespoke wiring.

Picker shows alongside admin actions (does NOT replace them in the layout — visible below the action buttons row when open). When empty, an inline hint replaces the dropdown. Confirm button is disabled until a member is picked.

## 10. Decision log

| Decision | Why |
|---|---|
| New method `markMessageViewed(channelId, messageId)` rather than extending `markSeen(messageId)` to 2-arg | Existing tests (`toast-improvements.spec.js`, `prop-drilling.spec.js`) mock `markSeen = vi.fn()` as 1-arg; changing the signature would break them. New method keeps the contract clean and lets ChatView call BOTH from the same observer entry. |
| Back-compat `'listed'` in `availableChannels` filter | Existing test fixtures + older daemon emissions may still carry the legacy string. The filter accepts both; outgoing wire calls only use the canonical pair. Tightening the filter alone would break ~10 existing fixture-driven tests with no UX benefit. |
| Picker as native `<select>` not custom typeahead | Small member counts; better a11y; less code. |
| Snapshot `channelId` at observer-creation time (not at intersection time) | Mid-dwell channel switch should mark messages for the channel they were viewed IN, not the new channel. Cleanest implementation: $effect bound to `activeChannelAtObserve` rebuilds the observer on switch; in-flight timers retain the snapshotted id. |
| `DWELL_MS = 1000` as a module-level const | Pinned per spec ("≥ 1 second"); module const makes it discoverable + future-tunable without code search. |
| Mechanical literal updates in existing test files | The brief's "mechanical literal updates DO go in the new follow-ups spec file, not in existing files" was interpreted as "don't add NEW unrelated tests to old files." Where an existing test assertion was a direct regression on the OLD broken-relative-to-3.6b behavior (default visibility = `'listed'`, rename wire-shape = `name:`, Transfer click fires confirm), the assertion HAD to be updated to align with the now-correct behavior. The new follow-ups spec then layers explicit positive assertions on top. |
| Removed the stale `[VERIFY]` block in ChannelAdminPanel doc-comment | The 5 store accessors it called out as "do not exist yet" all landed in Wave B (3.6 + 3.6b). The block was outdated noise; replaced with the current Wave C picker docstring. |
