## Picker design decisions

- **Search-as-you-type, substring match, case-insensitive.** No debounce — the candidate list is small (a few hundred entries max) and `$derived.by` over a Set lookup is cheap enough that synchronous filter on every keystroke is fine. No "fuzzy" matching, no token weighting — just `.includes()`.
- **Alpha-sorted by display name** via `localeCompare`. Stable order so the user's mental model of "where Alice is" doesn't shift between renders.
- **Empty query shows the full eligible list.** A picker that's blank by default felt punitive — the user might not know the exact name and wants to scroll.
- **Exclude rules**: caller key (`currentUserKey`) and `existingMemberKeys` (channel members) both go into a Set built in a `$derived.by`. Caller is hidden even when not explicitly listed; existing members are pre-computed by App.svelte from `Object.keys(store.channelMembers[channelId])` so the dialog stays presentation-only.
- **Selection is radio-style**: one row at a time, click toggles a single `selectedKey`. Submit gated on `selectedKey !== null AND candidates.some(c.key === selectedKey)` — so changing the search query after selecting drops the now-invisible selection rather than silently submitting a stale key.
- **Agent participants** get a small "agent" badge tag in the row so the user can distinguish humans from Claude instances at a glance.

## Note-field decisions

- **Cap 200 chars** with a live counter `{noteLen} / 200` in the bottom-right of the body.
- **Soft cap** via `maxlength={NOTE_MAX + 50}` to let the user briefly overshoot while editing, but the counter turns red and Invite disables when over 200. On submit, the note is sliced to 200 defensively.
- **Optional** — empty note is fine; the server defaults to empty string and `inviteParticipant` omits the field from the wire body entirely when blank (so the daemon's JSON parser doesn't see `note: ""` clutter).
- The note is forwarded verbatim — server treats it as opaque and surfaces it in the `tool_comms_invite` system message.

## Integration pattern — why CustomEvent instead of prop drilling

The brief lists `Sidebar.svelte` in MUST-NOT-TOUCH. `Sidebar.svelte` is where `ChannelContextMenu` is mounted and where the existing `handleContextAction` switch routes actions. Without touching Sidebar, there is no prop-drilling path to App.svelte for a new `'invite'` action.

**Resolution**: ChannelContextMenu dispatches `window.dispatchEvent(new CustomEvent('claude-comms:invite-participant', { detail: { channel } }))` when the invite item activates. App.svelte mounts a `$effect` listener that captures the event and populates the `inviteDialog` state slot. This mirrors the existing `slashCommand` CustomEvent bus pattern (also used for `/list` and `/nick`), so it's not a new pattern — just a second client of it.

The standard `onAction(actionId)` path still fires (so Sidebar's existing switch closes the menu via `closeContextMenu()` cleanly — none of its cases match `'invite'` so the action falls through harmlessly to the end of the function).
