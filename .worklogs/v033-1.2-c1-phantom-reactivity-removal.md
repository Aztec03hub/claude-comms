## §1 Scope

Remove the two cargo-cult workarounds added against a phantom Svelte 5 reactivity bug that does not reproduce per the principal-engineer review with MCP-doc validation. Class-based `$state` properties on stores ARE fully reactive across component boundaries when consumed via `$derived`. Replaced the 100 ms `flushSync` polling pump in `ChatView.svelte` with a single idiomatic `$derived`; deleted both `this.messages = this.messages` self-assignment kicks in `mqtt-store.svelte.js` (and the doc-comment line that advertised the pattern).

## §9 Outstanding concerns

1. **Minor judgment call:** removed one extra doc-comment line (`* Triggers reactivity via self-assignment.`) at the `toggleReaction` JSDoc because keeping it would falsely document a behavior that no longer happens. The spec said "delete any adjacent comment like `// kick reactivity`" — I interpreted the JSDoc line as adjacent in spirit (it advertises the same pattern). If orchestrator prefers stricter scope, this is a one-line revert in 30 seconds.

2. **The fallback expression** `(messagesProp ?? [])` is technically defensive: `messagesProp = []` is already the prop default in `$props()` destructuring, so `??` is only relevant if a caller explicitly passes `null` or `undefined`. Kept it to match the original `messagesProp` semantics rigorously (the old pump used `source && source.length !== _lastSyncLen` which guarded against null). Costs nothing at runtime.

3. **App.svelte still passes `messages={store.activeMessages}` AND `{store}`** to `<ChatView>`. With the new `$derived(store ? store.activeMessages : messagesProp)`, the `messages` prop is effectively dead when `store` is present. This is a no-op (the `$derived` picks the store branch), but future cleanup could drop the prop from the App.svelte call site. Out of scope for Step 1.2 (App.svelte is not in the writeable set).

4. **The "Observe only NEW message elements as they appear" effect** at lines 96–112 of the new file uses `const _len = messages.length;` to trigger re-reactivity on length changes — this is now unambiguously sound since `messages` is `$derived` from `store.activeMessages` and Svelte's reactivity graph will re-run the effect whenever the store mutates. Previously this worked only because the pump's local `$state` was being reassigned every 100 ms. Verified by visual reasoning + the test-suite green.
