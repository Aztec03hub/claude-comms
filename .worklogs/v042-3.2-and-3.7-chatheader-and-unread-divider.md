## 3. ChatHeader inline-edit interaction details

- **Enter** â†’ call `commitEditTopic()`. Snapshot the draft into a local
  `nextTopic`, then immediately clear `editingTopic = false` and
  `topicDraft = ''` BEFORE the awaited `store.setTopic(...)` resolves.
  This prevents the auto-blur fired by Svelte re-rendering past the
  `<input>` from re-entering `commitEditTopic()` while the previous call
  is still in flight.
- **Esc** â†’ `cancelEditTopic()`: zero the draft, exit edit mode, no
  store call. `e.stopPropagation()` so the dialog-Esc handlers higher up
  the tree (none exist for ChatView today but the wrapper-modal pattern
  is common) don't double-fire.
- **Blur** â†’ if `editingTopic` is still true (i.e. user clicked away
  rather than pressing Esc), commit. Matches `ChannelDirectoryModal`
  Admin tab.
- **Same-value no-op**: if the draft equals the current channel topic,
  the store call is skipped (saves an MQTT roundtrip).
- **Empty string accepted**: `setTopic(id, '')` is allowed and forwarded
  to MCP. The store's existing `comms_conversation_update` accepts empty
  strings; this matches user intent "clear the stale topic."
- **Optimistic update + rollback**: handled inside `store.setTopic`
  itself (mqtt-store.svelte.js:2142-2165 â€” no work needed here).
- **Error surfacing**: when `store.setTopic` resolves
  `{ success: false, error }`, ChatHeader fires the optional
  `onEditTopicError(error)` callback. ChatView forwards this via
  `onTopicEditError` to its caller. We do NOT pipe directly into
  notifications.svelte.js â€” that is the parent's job (keeps ChatHeader
  test-isolatable and matches ChannelDirectoryModal's silent-rollback
  pattern when no error sink is provided).
- **Role gating**: `canEditTopic = $derived(currentUserRole === 'owner'
  || currentUserRole === 'admin')` drives both the pencil-button render
  AND the static-button `disabled` state AND its `class:editable` class.
  When `canEditTopic` is false, `startEditTopic` early-returns so clicks
  on the static button do nothing even if a future styling regression
  re-enables them.

## 4. UnreadDivider positioning logic

The store already exposes per-channel `unreadFrom` (the message id of the
first unread message) via `channelsById[id]` â€” this is the legacy v0.3.x
marker persisted in localStorage by `mqtt-store.svelte.js#saveUnreadMarkers`
and restored on bootstrap by `#restoreUnreadMarkers`. The unread count is
`channelsById[id].unread`.

ChatView's `groupedMessages` derivation now:

1. Builds the existing date/system/messages groups (unchanged).
2. If `unreadCursorId` is non-null AND `unreadCount > 0`, walks the
   `groups` array and splices an `{ type: 'unread-divider', count: N }`
   row in front of the FIRST group that contains a message with
   `m.id === unreadCursorId`.
3. The template `{#each groupedMessages as group, i (i)}` has a new
   `{:else if group.type === 'unread-divider'}` branch that renders
   `<UnreadDivider unreadCount={group.count} />`.

Why "first group containing the cursor" rather than "split the group at
the cursor": the message grouping is purely a visual consecutive-sender
optimization, so inserting the divider AT the message boundary is what a
user expects. We don't split a sender's run of messages just to slip a
divider mid-run â€” the divider sits between groups, just before the run
that starts the unread window.

Hidden cases:
- `unreadCursorId === null` â†’ no divider.
- `unreadCount === 0` â†’ no divider (even if cursor is set; pin in
  test #3 of `ChatView â€” UnreadDivider positioning`).
- Cursor points at a message not in the current `messages` array (older
  history not paged in) â†’ loop exits without splicing â†’ no divider. Pin
  in test #2.
