## 1. WHAT shipped

Two coupled deliverables landed in this step:

1. **Frontend store accessors** (scope expansion baked in by the orchestrator's pre-dispatch §I.18 audit step 1.5):
   - `store.setProfileStatus(emoji, text, expiresAt = null)` — wraps Wave A2's `comms_profile_status_set` MCP tool.

   Both follow the **Wave-B admin-accessor pattern** (mirror of `setTopic` / `deleteChannel`): optimistic local mutation of `userProfile.profileStatus`, MCP HTTP call via `mcpCall(...)`, snapshot-restore on failure, `{success, error}` return envelope for the caller to wire to a toast.

2. **`StatusEditor.svelte` popover** + Sidebar status row:

   - Inline 8-emoji strip (`💬 🍵 🎧 🧠 🛌 🏃 🤒 🌴`) + free-text emoji input (so any glyph the user pastes works).
   - 60-char text input with live char counter that goes red on overflow.
   - 4 expiry presets: **Never / 1 hour / 4 hours / Until tomorrow** (the spec's 5th "Custom" datetime picker is deferred — see Trade-offs §5).
   - Save / Clear / Cancel buttons.
   - Sidebar status row sits in the existing `.user-profile` block under the connection-status line: `<emoji> <text>` if set, italic "Set a status" placeholder otherwise. Click opens the popover.

## 3. SidebarFooter vs inline — DECISION: **inline**

The orchestrator audit (§I.18 step 1.5) flagged that `SidebarFooter.svelte` does NOT exist; the identity row is inline at `Sidebar.svelte:204-235`. Two options were offered:

- (a) **Extract** the identity row + StatusEditor mount into a new `SidebarFooter.svelte`.
- (b) **Inline** the StatusEditor mount alongside the existing identity row.

**Chose (b) — inline.** Rationale:

1. The existing identity-row block at `Sidebar.svelte:204-235` is **tightly coupled** to local `$derived` state (`connectionLabel`, `connectionState`) used in both the `<div class="status-dot">` and `<div class="ustatus">` lines. Extraction would require either prop-drilling those two derived values through a new child or re-deriving them inside the new component (duplication).
2. The G-25 connection-status footer test coverage lives in `sidebar-fixes.spec.js` and uses the inline DOM hierarchy; an extract would force a refactor of that adjacent test surface — unnecessary scope creep.
3. The status row is **one button** + **one conditional popover** — too small to warrant its own component yet. If a future step adds more footer surfaces (e.g. role indicator, mention badge), revisit then.

Net result: 59 lines added to `Sidebar.svelte`, 0 new component files created for the footer concern.

## 4. EmojiPicker reuse vs inline — DECISION: **roll inline**

The existing `EmojiPicker.svelte` is a full-screen modal:
```css
.emoji-backdrop { position: fixed; inset: 0; z-index: 99; }
.emoji-picker { position: fixed; bottom: 120px; left: 50%; transform: translateX(-50%); width: 380px; ... }
```

Mounting it from inside another popover would create a visual layering fight (two backdrops, two modals, ambiguous z-index, two Escape handlers). The 8-glyph strip rolled inline (`28×28` buttons + 56-px free-text fallback input) covers the status-set use-case in **18 lines of markup** + **34 lines of CSS** — net cheaper than wiring + re-styling EmojiPicker for the embedded context.

If a future ProfileCard / "react to your own status" feature ever needs the full picker inside a status flow, revisit.

## 5. Trade-offs / deviations baked in

- **"Custom" expiry preset deferred.** The spec lists `Never / 1h / 4h / Until tomorrow / Custom`. A native `<input type="datetime-local">` works in some browsers but renders inconsistently and lacks timezone affordances; a custom datetime picker is out-of-scope for this step. The 4 baked-in presets cover the common cases (afk, lunch, end-of-day, "tomorrow"). Documented in StatusEditor.svelte's top-of-file comment.
- **"Until tomorrow" = 24 hours from now**, NOT literal next-midnight. Keeps the expiry boundary timezone-independent and matches how a user reads "until tomorrow" colloquially ("I'll be back ~this time tomorrow").
- **Disconnected-state behaviour.** Brief said "queued via `pendingSends` pattern (mirror Wave B's accessors)." Mirror-of-Wave-B is the operative phrase — Wave B's `setTopic` / `deleteChannel` / `renameChannel` accessors do NOT actually use `#pendingSends` (that queue is reserved for outgoing MQTT chat messages per UX G-62). They fire the MCP HTTP request unconditionally and return `{success, error}`. The new accessors follow that exact pattern. If the orchestrator intended literal MCP queuing, surface and I can rework — but the existing Wave B precedent is unambiguous.
