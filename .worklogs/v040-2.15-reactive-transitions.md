## 1. Scope

Add the 4-phase reactive transition system per Design Spec §10. When the
mqtt-store's lifecycle methods (Step 2.6) move a channel between sidebar
sections (join → into Active; leave → out of Active; star → into
Starred; close → out of all sections), the section component now plays
the 4-phase choreography:

  1. **Phase 1 — fade-out from old section** (~225ms) — `channelFlyOut`
  2. **Phase 2 — gap-collapse in old section** (~225ms) — folded into `channelFlyOut`
  3. **Phase 3 — gap-grow in new section** (~225ms) — folded into `channelFlyIn`
  4. **Phase 4 — fade-in to new section** (~225ms) — `channelFlyIn`

Total budget ~900ms door-to-door (the destination-side phases overlap
with the source-side phases, so perceived dead time is ~450ms).
`prefers-reduced-motion: reduce` snaps to instant (0ms duration).

## 4. Design decisions

### 4.1 Composition over coordination

The orchestrator's spec sketched an imperative coordinator function
`transitionChannel(fromSection, toSection, channelId)` that the store
would call to drive all four phases. I rejected this in favor of
**composing Svelte's built-in transition system declaratively**:

  - When the store moves a channel between section arrays, Svelte's
    keyed `{#each}` already detects the row vanished from one section
    and appeared in another.
  - The vanishing row plays `out:rowFlyOut` (phases 1+2); the
    arriving row plays `in:rowFlyIn` (phases 3+4). Both fire in
    parallel because the store mutates both arrays in the same tick.
  - No imperative coordinator is needed — Svelte handles the phase
    sequencing implicitly.

This is more Svelte-idiomatic and avoids a `setTimeout` chain that
would race the framework's own transition lifecycle.

### 4.2 Why height+opacity are combined into one transition

The initial attempt was to stack `transition:channelSlide` (height)
alongside `in:channelFlyIn` / `out:channelFlyOut` (opacity + translate)
on the same wrapper, so each effect would have its own clean primitive.
The Svelte autofixer caught this immediately:

```
Cannot use `transition:` alongside existing `in:` directive
https://svelte.dev/e/transition_conflict
```

Svelte forbids mixing the bidirectional `transition:` with the
unidirectional `in:`/`out:` on the same element. Options:

  a) Drop `in:`/`out:` and use `transition:` for everything — but
     then we can't distinguish entering vs. leaving in the css()
     callback, because Svelte's `transition:` re-uses the same
     function for both directions and reverses `t`.
  b) Drop `transition:` and fold the height-collapse math INTO the
     `in:`/`out:` directives — i.e., have `channelFlyIn` and
     `channelFlyOut` animate height/padding/margin/border alongside
     opacity and transform. (Same trick Svelte's built-in `slide`
     uses internally.)
  c) Nest two divs: outer carries `transition:slide`, inner carries
     `in:fly`/`out:fly`. Works but doubles the DOM.

I chose **(b)** — `channelFlyIn` and `channelFlyOut` measure the
node's computed height + paddings + margins + borders at transition
kickoff, then interpolate them all to 0 (or back) alongside opacity
and translateY. One directive, one DOM element, all four phases.

### 4.3 `transitionFlavor` prop instead of a single transition function

The section now accepts `transitionFlavor: 'fly' | 'crossfade' | 'instant'`:
  - `'fly'` (default) — full 4-phase choreography for cross-section moves
  - `'crossfade'` — smaller (150ms) opacity-only crossfade for star toggles
  - `'instant'` — no animation, for testing or for downstream sections
    that want a different feel (e.g., search-result panels in 2.19).

The flavor lives at the SECTION level, not the row level, because
which animation plays depends on WHICH section the row enters/leaves,
not on the row itself. The row receives the flavor as a `data-*`
attribute for inspection only — it doesn't mount its own transitions.

### 4.4 Reduced motion: defense in depth

Two layers:
  1. **Function level** — every transition function in
     `channelMove.svelte.js` short-circuits to `{duration: 0}` when
     `prefersReducedMotion()` returns true. This is the primary defense.
  2. **CSS level** — `.sidebar-channel-section-row-wrap` carries an
     additional `@media (prefers-reduced-motion: reduce) { transition:
     none !important; }` rule. This catches the edge case where
     `matchMedia` reports stale (some browsers debounce media-query
     callbacks under heavy load).

The "no surprise" rule (§10.8) is preserved: rows still mount/unmount
in the right slot under reduced motion, just without animation.

## 7. Known limitations / handoffs

- The transitions module DOES NOT yet expose an imperative
  coordinator (`transitionChannel(from, to, id)`). If a future step
  needs to manually trigger an animation outside the natural store
  → section reflow (e.g., the omnibar in Step 2.19 hand-animating a
  result-selection preview), it can build that on top of the
  primitives this module exports. Out of scope for v0.4.0 MVP.
- The `starToggle` transition currently only animates opacity (not
  position). A more elaborate "row crossfades up to Starred section"
  effect would need access to both the source and destination
  bounding rects — that's a Svelte `crossfade()` use case (the
  `send`/`receive` pair). Punted to v0.5.x if Phil wants a more
  cinematic star flow.
- `channelSlide` is exported but not used inside this step's
  components — it's there for the close (archive/delete) flow,
  which Step 2.10's context-menu wiring may drive.
- The row wrapper adds one extra DOM element per channel. Negligible
  at the typical sidebar scale (~20 channels) but worth noting if
  someone profiles the sidebar with hundreds of channels.
