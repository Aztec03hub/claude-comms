Handled types (rows below in the order they appear in the switch):

| Type | Daemon source | Effect |
|---|---|---|

| `topic_changed` / `conversation_topic_changed` | `comms_conversation_update` (v0.3.2) | Update `channels[id].topic` |

| `unarchived` | `comms_conversation_unarchive` (Step 2.3) | Flip `archived = false` + clear archive provenance |
| `member_joined` | forward-compat — backend not yet publishing on system topic | Bump `memberCount` + `lastActivity`; flip `member = true` for self |
| `member_left` | forward-compat | Decrement `memberCount` (clamped ≥ 0); flip `member = false` for self; auto-unstar |
| any other | unknown — log + bump parse-failure rate + skip; NEVER throw |

Two extension fields the brief asked for:

- **Dual id-field acceptance**: events from Steps 2.2 / 2.3 carry `msg.id`,
  v0.3.2 originals carry `msg.name`. New code prefers `msg.id`, falls
  back to `msg.name`, logs + skips if neither is present.
- **Lifecycle toast**: new `latestChannelLifecycleToast = $state(null)`
  on the store, populated by a single private `#emitChannelLifecycleToast`
  helper that respects the existing `inAppToasts` setting. App.svelte
  consumers can subscribe via a `$effect` in a future step.

## 4. Code change — handler rewrite (representative excerpt)

```js
#handleSystemConversation(msg) {
  if (!msg || typeof msg !== 'object' || typeof msg.type !== 'string') {
    this.#logSystemConversationParseFailure('non-object or missing type', msg);
    return;
  }
  // Dual-format id resolution — Steps 2.2 / 2.3 events carry ``id``,
  // v0.3.2 events carry ``name``. Prefer id; fall back to name.
  const id =
    typeof msg.id === 'string' && msg.id.length > 0
      ? msg.id
      : typeof msg.name === 'string' && msg.name.length > 0
        ? msg.name
        : null;
  if (id === null) {
    this.#logSystemConversationParseFailure('missing both id and name', msg);
    return;
  }
  switch (msg.type) {
    case 'created':
    case 'conversation_created': { /* full ChannelRow populate */ }
    case 'topic_changed':
    case 'conversation_topic_changed': { /* update topic */ }
    case 'renamed': { /* update name (forward-compat) */ }
    case 'deleted':
    case 'conversation_deleted': { /* remove + clear buffer + switch + toast */ }
    case 'archived': { /* flip + stamp + clear buffer + switch + toast */ }
    case 'unarchived': { /* flip back, clear archive provenance */ }
    case 'member_joined': { /* bump count + self-member flip */ }
    case 'member_left': { /* decrement (clamped) + self-leave auto-unstar */ }
    default: this.#logSystemConversationParseFailure(
      `unknown type ${JSON.stringify(msg.type)}`, msg,
    );
  }
}
```

The defensive guard at the top of the function rejects three failure
classes BEFORE the switch:

1. `null` / non-object / missing `type` → log + skip.
2. Both `id` and `name` absent → log + skip.
3. Unknown `type` → log + skip (handled by the `default` case).

In all three cases the parse-failure counter is bumped so the
App.svelte banner can surface a spike if multiple bad payloads land in
quick succession (≥ 5 in 30s threshold per v0.3.1).

## 5. The "user is currently viewing this channel" switch logic

For both `deleted` and `archived`, the switch follows this order:

1. Snapshot `wasActive = (this.activeChannel === id)` BEFORE any
   mutation so the post-mutation toast can branch correctly.
2. Snapshot `removedRow` / `existing` for the toast's `channelName`
   field (the row's display name, not the routing id).
3. Apply the mutation (`delete` for deleted; `archived = true` + stamps
   for archived).
4. Filter `this.messages` to drop messages with `channel === id` —
   matches the same pattern used by `leaveChannel`'s commit path at
   line 1947.
5. If `wasActive`: null out `activeChannel`, then call
   `#resetActiveChannelIfStale()` which picks the first alpha-sorted
   member channel, or `null` if none.
6. Emit the toast — gated on either `removedRow` (some local row was
   actually affected) or `wasActive` (the user noticed even if our
   local map didn't have the row).

For `unarchived` there is no toast and no message-buffer clear — the
operation is non-destructive and the row simply re-appears in
`availableChannels` via the `$derived` projection's archive filter.
