## 1. What shipped

1. **`store.checkChannels()`** , single `comms_check` MCP call (no `conversation` arg → server scans every joined conv) that hydrates `channels[id].unread`, `channels[id].lastActivity`, `channels[id].unreadHasMention` from the response. Zeroes unread on joined channels NOT surfaced in the response (server-authoritative invariant).
2. **`store.getChannelRole(channelId)`** , client-side inference returning `'owner' | 'member' | null`. Owner when `channel.createdBy === userProfile.key` OR (legacy grandfather) `=== userProfile.name`. Writes to reactive `channelRoles` $state cache.
3. **`store.renameChannel(channelId, newName)`** , optimistic local update + MCP `comms_conversation_update` with `name` field + rollback on error + queue when disconnected.
4. **`store.setVisibility(channelId, level)`** , same pattern with `visibility` field. Accepts panel-driven values (`'public'`/`'private'`) verbatim; backend extension will accept new value space without client change.
5. **`store.setMode(channelId, mode)`** , same pattern with `mode` field.
6. **`store.transferOwnership(channelId, newOwnerKey)`** , same pattern with `created_by` field. 1-arg call (from current ChannelAdminPanel without picker) returns structured error envelope. Also demotes `channelRoles[id]` to `'member'` on success.
7. **`document.visibilitychange` listener** , installed in connect handler, detached in disconnect. Calls throttle-aware `#maybeCheckChannels()`.
8. **30s throttle** , `#COMMS_CHECK_THROTTLE_MS` static; `#maybeCheckChannels()` short-circuits with `{ throttled: true }` envelope when within the window. Connect handler resets the slot to 0 so the initial fetch always runs.
9. **`#pendingAdminActions` queue** , parallel to `#pendingSends` but holds MCP RPC closures (`run` + `rollback`). Drained on the same `'connect'` event. Cap=50, drop-oldest with rollback fires.

## 7. Client-side inference fallback for `getChannelRole` + [VERIFY]

**Decision:** For Wave B, `getChannelRole` returns role via CLIENT-SIDE INFERENCE only:
- `channel.createdBy === userProfile.key` → `'owner'`
- `channel.createdBy === userProfile.name` (3.0a grandfather backfill for pre-schema-v2 rows) → `'owner'`
- Any other case → `'member'`
- `'admin'` is never synthesized client-side.

**Rationale:** v0.4.2 Step 3.0a (`e06ebdf`) shipped the backend `RegistryStore.get_channel_role` method but did NOT add an MCP wrapper exposing it (verified by grep against `src/claude_comms/mcp_tools.py` + `mcp_server.py` , no `comms_get_channel_role` tool exists). The store cannot query the authoritative role without that wrapper. Falling back to inference matches the brief's orchestrator-pinned decision and keeps the role-gated UI usable today.

**[VERIFY] surfaced for follow-up step:** Add `comms_get_channel_role` MCP wrapper exposing 3.0a's `RegistryStore.get_channel_role` so the store can hydrate `channelRoles` async on bootstrap + channel-join. Wave C or a small Wave B.5 step can land this; once it lands, `getChannelRole` switches to async with cache-then-fetch semantics. The reactive `channelRoles = $state({})` cache is already plumbed so existing consumers (ChannelAdminPanel via prop, future MemberContextMenu) will re-render without code changes.

## 10. Decision log (pragma calls)

1. **Separate queue for admin actions, NOT reuse of `#pendingSends`.** The existing queue holds `{ messageId, topic, payload }` snapshots for MQTT broker publishes; admin actions are MCP RPC closures with their own rollback callbacks. Same drain hook (`'connect'`) but a separate queue keeps the shapes honest and the drain logic per-queue. Mirrors the brief's "Mirror exactly , same queue" intent in spirit (same hook, same FIFO discipline, same drop-oldest cap, same drain-on-reconnect) while keeping the shape obvious. Decision logged in `#pendingAdminActions` field doc.
2. **Optimistic local update fires even when disconnected.** This makes the UI feel correct immediately and rollback engages on drain-failure. Alternative (reject all when disconnected) would degrade UX without obvious benefit.
3. **`getChannelRole` is sync, not async.** The reactive cache pattern means consumers can subscribe via `$state` reactivity; making the accessor async would force callers into effect-orchestration overhead for a value that is, today, derivable synchronously from local data. When the backend wrapper lands, the cache write happens out-of-band (in a separate async hydration helper) so the sync read surface stays stable.
4. **30s throttle is calibrated by the brief**; no decision needed.
5. **`checkChannels()` is the public name** (NOT `commsCheck`); chosen to match the brief's "preferred name" hint and because it's the action a UI/consumer would describe.
6. **Connect-handler resets `#lastCommsCheckAt = 0` BEFORE calling `checkChannels()`** so the initial fetch always runs even if a previous tab session had advanced the stamp. The throttle only protects the `visibilitychange` re-fire path.
