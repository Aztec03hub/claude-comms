## 1. Scope

Extend the `/api/conversations` HTTP endpoint to:

1. Return the **daemon's full known conversation set** (not just the caller's memberships) so the v0.4.0 web sidebar's "Available" section can populate from server truth rather than a hardcoded seed list.
2. Emit the v0.4.0 **ChannelRow** payload per Design Spec §13.4: `{ id, name, topic, member, memberCount, lastActivity, mode, visibility, createdAt, createdBy, myUnread, myStarred, myMuted }`.
3. Apply **visibility rules**: listed channels appear for every caller; unlisted channels appear only for callers who are members.
4. Preserve **backward-compat** snake_case fields (`joined`, `member_count`, `last_activity`, `created_at`, `created_by`, `message_count`) on every row so v0.3.x web clients keep working through the v0.4.0 rollout.

Non-goals (deferred):

- Persisted `mode`/`visibility` on `ConversationMeta` — currently defaulted to `"public"`/`"listed"` via `getattr`. Flagged in §9.
- Real `myUnread`/`myStarred`/`myMuted` values — v0.4.1 will populate these when per-user state lands.

### 4.1 `src/claude_comms/mcp_server.py` — appended (after line 399)

```python
# ---------------------------------------------------------------------------
# v0.4.0 full ChannelRow serialization for /api/conversations
# ---------------------------------------------------------------------------
#
# Design Spec §13.4 ChannelRow contract:
#   { id, name, topic, member, memberCount, lastActivity,
#     mode, visibility, createdAt, createdBy,
#     myUnread, myStarred, myMuted }
#
# Backward-compat fields preserved alongside (snake_case "joined",
# "member_count", "last_activity", "created_at", "created_by",
# "message_count") so v0.3.x web clients keep working during rollout.


def _serialize_conversation_full(
    meta: Any,
    caller_key: str,
    *,
    registry: Any = None,
    store: Any = None,
    activity_tracker: Any = None,
) -> dict:
    """Serialize a ConversationMeta into the v0.4.0 ChannelRow shape.

    *caller_key* is the calling identity's 8-hex key. When empty, ``member``
    is always ``False`` and unlisted channels MUST be filtered upstream.

    *registry*, *store*, *activity_tracker* default to the module-level
    globals when omitted (production path). Parameterized so unit tests
    can inject fakes without monkey-patching module state.
    """
    reg = registry if registry is not None else _registry
    msg_store = store if store is not None else _store
    tracker = activity_tracker if activity_tracker is not None else _activity_tracker

    # caller membership + total member count
    is_member = False
    member_count = 0
    if reg is not None:
        members = reg.members(meta.name)
        member_count = len(members)
        if caller_key:
            convs = reg.conversations_for(caller_key)
            is_member = meta.name in convs

    # last activity: tracker → meta.last_activity → meta.created_at
    last_activity = None
    if tracker is not None:
        last_activity = tracker.get(meta.name)
    if not last_activity:
        last_activity = meta.last_activity
    if not last_activity:
        last_activity = meta.created_at

    # mode / visibility: defensive getattr so a future field-add transparently surfaces
    mode = getattr(meta, "mode", None) or "public"
    visibility = getattr(meta, "visibility", None) or "listed"

    message_count = 0
    if msg_store is not None:
        message_count = len(msg_store.get(meta.name))

    return {
        # ChannelRow (v0.4.0)
        "id": meta.name,
        "name": meta.name,
        "topic": meta.topic,
        "member": is_member,
        "memberCount": member_count,
        "lastActivity": last_activity,
        "mode": mode,
        "visibility": visibility,
        "createdAt": meta.created_at,
        "createdBy": meta.created_by,
        "myUnread": 0,
        "myStarred": False,
        "myMuted": False,
        # back-compat (v0.3.x) — keep until v0.5.x deprecation
        "joined": is_member,
        "member_count": member_count,
        "last_activity": last_activity,
        "created_at": meta.created_at,
        "created_by": meta.created_by,
        "message_count": message_count,
    }


def get_all_conversations_full(caller_key: str = "") -> list[dict]:
    """Daemon's full known conversation set, ChannelRow-shaped.

    Visibility rules:
      - Listed channels (visibility == "listed") appear for everyone.
      - Unlisted channels appear only for callers who are members.
    """
    if _conv_data_dir is None:
        return []
    metas = list_all_conversations(_conv_data_dir)

    caller_memberships: set[str] = set()
    if _registry is not None and caller_key:
        caller_memberships = set(_registry.conversations_for(caller_key))

    result: list[dict] = []
    for meta in metas:
        visibility = getattr(meta, "visibility", None) or "listed"
        is_member = meta.name in caller_memberships
        if visibility != "listed" and not is_member:
            continue
        row = _serialize_conversation_full(meta, caller_key)
        result.append(row)
    return result
```

## 7. Findings surfaced

1. **`ConversationMeta` lacks `mode`/`visibility`** — both default to `"public"`/`"listed"` via `getattr`. This is intentional per the spec ("default `\"public\"` if not tracked yet — flag as follow-up"), but means until the meta schema migration lands, every channel will appear `mode="public", visibility="listed"`. The unlisted/private tests use a runtime attribute injection via `object.__setattr__` (allowed because `ConversationMeta` isn't frozen). See §9 outstanding item 1.

2. **`?all=true` query param is now a no-op** — the endpoint always returns the full set (visibility-filtered per caller). The param is still accepted (back-compat) but logged-out for future deprecation. v0.4.x web client shouldn't rely on it.

3. **`get_all_conversations` (the v0.3.x function) is left in place** — not deleted yet because external Python callers may import it. Marked as deprecation candidate for v0.5.x. No callsites remain inside this repo (verified with `grep -rn "get_all_conversations\b" src/ tests/`).

4. **Test PYTHONPATH workaround documented** — the worktree shares the parent's editable-install venv, so testing requires `PYTHONPATH=$PWD/src` to override the parent's `src/`. This is a worktree-isolation friction point that will recur for every backend agent in Wave A; orchestrator may want to document in §I.16.

5. **Skipped test `test_index_html_exists_when_web_dir_present`** — exists in the parent's count but the worktree has no built `web/dist/` bundle. Net `+19` from parent baseline 1207 → worktree observed 1225 passed (+ 1 skip the parent doesn't have). Re-running in the integrated branch after `pnpm build` will restore that test.

## 9. Outstanding concerns

1. **`mode`/`visibility` follow-up ticket** — `ConversationMeta` needs first-class fields for these in v0.4.x. Suggested migration: add `mode: Literal["public", "private"] = "public"` and `visibility: Literal["listed", "unlisted"] = "listed"` to `ConversationMeta` (with `populate_by_name=True` so existing meta.json files without those keys still validate). Then surface them in `comms_conversation_create` and the conversation-events stream. Until then, every channel reads as `public`/`listed` from this endpoint, which makes the unlisted/private code path inert in production. The collector's filter is correct; only its inputs are not yet differentiated.

2. **`myUnread`/`myStarred`/`myMuted`** — placeholders (`0`, `false`, `false`) per spec. v0.4.1 needs a per-user state store (per-key cursors for unread; per-key prefs map for starred/muted). Until then the sidebar will show zero unread badges and no star/mute UI state.

3. **Spec §13.4 list vs implemented** — Design Spec §III.4 preamble (line 955) mentions `starred`, `muted`, `unread`, `unreadHasMention` (no `my` prefix) as ChannelRow fields. This step's brief (line 992-999) uses `myUnread`, `myStarred`, `myMuted` with the `my` prefix. **I followed the brief's prefixed naming** because it's the more recent and more specific instruction, but flagging for orchestrator review — if §III.4 is canonical, the store's `$derived` projections may need to read both keys for transition. Not blocking for 2.5/2.6 (svelte store) since the store author can pick either name and consume the chosen field.

4. **`lastActivity` resolution** — currently per-conversation, not per-message. If a high-traffic channel has a new message every 100ms, the tracker's batched 5s flush means `lastActivity` can lag by up to 5s. For sidebar sort order this is fine (the tracker's in-memory `get()` is consulted first, so the lag is bounded to one tracker tick). Documented for completeness — not a defect.

5. **No HTTP-level integration test in `test_api_conversations.py`** — I considered standing up a TestClient (`from starlette.testclient import TestClient`) wrapped around a hand-built ASGI app, but the existing `_api_conversations` handler is defined inside `cli.start()`'s closure scope (closes over `config` + `_cors`), so testing it via TestClient requires factoring the handler out OR re-creating the closure. The collector-level round-trip test (#19) covers the same payload contract that a TestClient hit would, and the broker-level smoke test will be exercised by the orchestrator's `curl http://localhost:8788/api/conversations` spot-check after integration to main. If a fuller HTTP-level test is desired, it should be a separate refactor step (extract `_api_conversations` to a module-level function taking `config`/`_cors_fn` as args).

6. **PYTHONPATH friction in worktree isolation** — see Finding 4. Worth a §I.16 addendum: "agents working in a worktree must export `PYTHONPATH=$PWD/src` before running pytest, OR re-install the editable package against the worktree (slower). The orchestrator's parent-tree integration verification will use the editable install naturally."
