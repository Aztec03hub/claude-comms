## 1. What shipped

A POST `/api/invite` REST endpoint that bridges to the existing
`tool_comms_invite` MCP tool. The endpoint is unconditionally registered
on the daemon (no feature flag), is CORS-covered by the v0.4.1
`CORSMiddleware` wrap that already encloses the MCP Starlette app, and
carries an explicit OPTIONS preflight route so the per-handler CORS
headers match every other `/api/*` endpoint in cli.py.

Body schema (matches Step 3.4 spec verbatim):

```json
{ "conversation_id": "...", "invitee_key": "...", "note": "..." (optional) }
```

Success response:

```json
{ "invited": true, "invitee_key": "...", "conversation_id": "..." }
```

Idempotency response (re-invite of an already-member): HTTP 409, body
`{ "invited": false, "reason": "already_member", "invitee_key": ..., "conversation_id": ... }`.

Side effects (system message on `claude-comms/conv/general/messages`)
belong entirely to `tool_comms_invite` — the REST handler is a surface
layer.

## 4. CORS verification

The `/api/invite` route is double-protected:

1. **Per-handler CORS headers** via `_cors_headers(...)` on every JSONResponse, matching the pattern in every other `/api/*` handler in cli.py.
2. **Starlette `CORSMiddleware` wrap** added in the v0.4.1 hotfix — applies to the entire MCP Starlette app at startup (lines 1149-1173 in cli.py, unchanged by this step). The middleware already lists `"POST"` in `allow_methods` and `"Content-Type"` in `allow_headers`, so the invite route is fully covered without modification.

Tests #12 and #13 (`test_invite_options_preflight_*`) prove the
per-handler path. The CORSMiddleware path is exercised by the existing
test_artifact_post_endpoint.py and test_cors_strict_mode.py suites,
which apply equally because the middleware is route-agnostic.

A curl-style preflight (post-daemon-start) would look like:

```bash
curl -i -X OPTIONS http://127.0.0.1:9920/api/invite \
  -H "Origin: http://127.0.0.1:9921" \
  -H "Access-Control-Request-Method: POST" \
  -H "Access-Control-Request-Headers: Content-Type"
# Expected: 200 + Access-Control-Allow-Origin: http://127.0.0.1:9921
#                + Access-Control-Allow-Methods: POST, OPTIONS
#                + Access-Control-Allow-Headers: Content-Type
```

A live POST would look like:

```bash
curl -i -X POST http://127.0.0.1:9920/api/invite \
  -H "Content-Type: application/json" \
  -d '{"conversation_id":"general","invitee_key":"deadbeef","note":"join us"}'
# Expected on success: 200 + {"invited":true,"invitee_key":"deadbeef","conversation_id":"general"}
# Expected when daemon-identity is not a member: 403 + {"error":"Caller ... is not a member ..."}
```

## 5. [VERIFY] items resolved / surfaced

**Resolved:**

- `comms_invite` MCP tool exists. Verified at `src/claude_comms/mcp_tools.py:1967-2019`, signature `(registry, publish_fn, *, key, conversation, target_name, message="", conv_data_dir)`, returns `{"status": "invited"}` / `{"status": "already_member"}` / `_error(msg)`. Tool is in scope for v0.4.x and NOT being modified by Step 3.4.
- REST CORS coverage. The v0.4.1 `CORSMiddleware` wrap already covers POST + OPTIONS for the entire MCP Starlette app, so the invite route inherits coverage. Per-handler `_cors_headers(...)` is additionally applied for parity with the rest of `/api/*`.
- The `invitee_key` -> `target_name` adapter. `tool_comms_invite` takes a *name* (via `registry.resolve_name`), but the REST spec mandates `invitee_key`. The handler looks up the participant by key via `registry.get(invitee_key)` and passes `participant.name` to the tool. Unknown invitee_key returns 400 BEFORE the tool is invoked.

**Surfaced (decision log, no blocker):**

- **Caller-identity policy.** The spec body does NOT carry a `caller_key`. The handler uses the daemon's configured `identity.key` as the inviter, consistent with how `/api/conversations` reads identity (cli.py line 1019). This makes the endpoint single-tenant for now: a browser session on the same machine as the daemon invites *as the daemon*, not as a separately authenticated user. Multi-tenant browser support is a v0.4.x+ follow-up (Step 3.14 territory). Recorded inline at cli.py with a `# Caller-identity policy:` comment block above the `build_invite_post_route` factory.
- **Idempotency choice: 409 vs 200.** Spec text said "pick one + test". Chose **409 with `{"invited": false, "reason": "already_member"}`** because (a) the success-shape requires `invited: true`, so a 200 no-op would be semantically lying, and (b) callers can branch on a non-2xx for re-invite UX without inspecting body fields. Test #11 locks the contract.
- **Note-field length / sanitization.** The spec doesn't bound `note` length and `tool_comms_invite` doesn't either. The handler validates type (must be str) but not length. If a future review wants a length cap, add it server-side in `tool_comms_invite` (out of scope here). Recorded for v0.4.x review.

## 6. Decision log

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Use the `build_X_route(...)` factory + provider-closure pattern instead of an inline `async def _api_invite` inside `_run_daemon` | Mirrors `build_artifact_post_route`. Lets tests mount the route into a Starlette app via `TestClient` without spinning up the broker. |
| 2 | Caller identity = `config.identity.key` (no body field) | Spec body schema doesn't include caller_key. Matches `/api/conversations` convention. Multi-tenant is Step 3.14 work. |
| 3 | Re-invite returns 409, not 200 | Avoids semantically-lying `{"invited": true}` on a no-op. Lets clients branch on status code without body inspection. |
| 4 | Invitee_key -> name adapter inside the handler (not pushed into the MCP tool) | `tool_comms_invite` is owned by Step 3.14. Adapter in REST layer keeps Step 3.4 surgical and avoids out-of-scope tool edits. |
| 5 | Unconditional route registration | Unlike artifact-POST (loopback + bearer + remote-edits flag), invite is a normal REST endpoint. No three-way matrix to test. |
| 6 | Per-handler `_cors_headers` AND outer `CORSMiddleware` | Same belt-and-braces as every other `/api/*` route in cli.py. Avoids divergence the next reviewer would flag. |
