## 1. Scope

After 5 failed reconnect attempts, the ConnectionStatus banner now stops
showing the indeterminate "Reconnecting…" line and instead surfaces
"Cannot reach broker" alongside two action buttons: **Retry now**
(invokes a parent-supplied `onRetry` callback wired to `store.connect()`)
and **Reload page** (calls `location.reload()`). The component owns its
own attempt counter — actually it reuses the pre-existing `retryCount`
that the error-state `$effect` already increments on every transition
into `(error truthy, connected false)`. The counter resets to 0 when
`connected` flips true, so a successful reconnect cleanly returns the
banner to its normal "Reconnecting…" indeterminate state on the next
failure cycle. The change is contained to one component and one new test
file; App.svelte wiring (`onRetry={() => store.connect()}`) is the only
follow-up needed and is called out in §7.

### 4c. Button onclick wiring

```diff
+  // UX G-27: "Retry now" button handler. Calls the parent-supplied callback
+  // (wired to `store.connect()` in App.svelte). The attempt counter is NOT
+  // reset eagerly — if the retry fails we want to stay in the failure state.
+  // Successful connection clears `retryCount` via the existing error effect
+  // (the `else if (connected)` branch sets it back to 0).
+  function handleRetry() {
+    if (typeof onRetry === 'function') {
+      onRetry();
+    }
+  }
+
+  // UX G-27: "Reload page" button handler. `location.reload()` is wrapped so
+  // tests can stub the location object. Guard against missing `location`
+  // (jsdom-without-window edge cases).
+  function handleReload() {
+    if (typeof location !== 'undefined' && typeof location.reload === 'function') {
+      location.reload();
+    }
+  }
```

  1. Indeterminate "Reconnecting…" banner while `retryCount` < threshold.
  2. Switches to actionable "Cannot reach broker" once threshold tripped

  3. Clicking **Retry now** invokes the `onRetry` callback exactly once.
  4. Clicking **Reload page** invokes `location.reload()` exactly once

     `reload` in place).

     the next failure cycle (verifies the visible state).
  6. After successful connect, the *counter* is actually reset (verified
     by `failureThreshold=2` cycle — error→connected→error leaves us at
     attempts=1, NOT failed; if the counter were sticky we'd be at
     attempts=2, failed).
  7. Default `failureThreshold=5`: drives 5 distinct error transitions
     (toggling the `error` prop string each rerender so the dependent
     `$effect` re-fires), confirms the banner flips into the failure
     variant on the 5th attempt.

## 7. Findings surfaced during work

- (S) **App.svelte wiring follow-up.** App.svelte is read-only per this
  step's scope, so the new `onRetry` prop is currently undefined in the
  production build — the button is rendered but is a no-op. Orchestrator
  should land a one-line edit on App.svelte (line 260):
  ```diff
  -<ConnectionStatus connected={store.connected} onlineCount={store.onlineCount} error={store.connectionError} />
  +<ConnectionStatus connected={store.connected} onlineCount={store.onlineCount} error={store.connectionError} onRetry={() => store.connect()} />
  ```
  Either as part of this step's commit (if scope is loosened) or as a
  trivial follow-up step. Without this, the user-facing impact of the
  Retry button is zero. The Reload button works regardless of wiring.
- (S) **Pre-existing `$effect`-with-untrack pattern surfaced 27
  autofixer "prefer $derived" suggestions.** All of these existed in the
  file before this step and pertain to the `retrySeconds` countdown and
  the connected-banner auto-hide timer. Both legitimately need
  imperative timer management (`setInterval`/`setTimeout` are stateful
  side effects, not pure derivations). Not worth refactoring in this
  scope. If the team ever wants to clean this up, the right move is to
  factor each timer into its own `.svelte.js` module that exposes a
  reactive value — out of scope here.
- (S) **`retryCount` semantics double up.** The same counter that
  drives the failure threshold is also used to gate the "Retrying in
  {N}s" countdown. This is intentional and correct (every error
  transition is both a "this is the Nth retry" event and a "show a 5s
  countdown" event), but if a future step ever needs to separate them
  (e.g. for finer-grained backoff), the rename would touch this file.

## 9. Outstanding concerns

- **Strict scope interpretation re: App.svelte.** I read "Files (write):
  web/src/components/ConnectionStatus.svelte" + the closing reminder
  "Do NOT touch any file outside the writeable scope" as forbidding
  App.svelte edits. The Done criterion "Retry button calls
  store.connect()" is satisfied by the component (when wired); the
  one-line App.svelte wire-up is documented in §7 as the trivial
  follow-up. If the orchestrator wants the wiring in this same commit,
  they can either add it themselves at commit time or bounce this
  worklog back with an expanded scope.
- **`failureThreshold` as a prop.** Spec says threshold = 5 and is
  silent on configurability. I exposed `failureThreshold` as a prop
  (default 5) primarily to keep the threshold-crossing tests
  deterministic (avoids fabricating 5 separate prop transitions in
  every spec). Production callers can ignore it. If the team
  considers this API surface bloat, it can be inlined as a `const`
  later — the test would then need to drive 5 transitions instead of 1
  (still feasible; spec #7 already does this).
- **`role`/`aria-live` flip on threshold.** When the banner flips into
  the failure variant I demote `role="alert"`/`aria-live="assertive"`
  to `role="status"`/`aria-live="polite"` (per the brief's a11y
  callout). Rationale: the user already has the actionable buttons in
  the page, so screen readers should not keep interrupting them
  every render. This matches the user's task switching from "wait
  for system" to "make a choice." Surfaced here because it's a
  judgment call not explicitly spelled out in the spec.
