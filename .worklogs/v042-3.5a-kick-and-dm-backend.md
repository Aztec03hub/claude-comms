## 3. DM slug algorithm

```python
def _dm_slug(key_a: str, key_b: str) -> str:
    lo, hi = sorted((key_a, key_b))
    return f"dm-{lo}-{hi}"
```

Python's default tuple sort on the two 8-hex-char strings is lexicographic, which is the natural lexicographic ordering for hex chars (`'0' < '9' < 'a' < 'f'`). The slug is therefore symmetric: `_dm_slug(A, B) == _dm_slug(B, A)` for any pair.

**Example:** caller key `"bbbbbbbb"`, target key `"aaaaaaaa"`:

```
sorted(("bbbbbbbb", "aaaaaaaa")) == ("aaaaaaaa", "bbbbbbbb")
slug = "dm-aaaaaaaa-bbbbbbbb"
```

The same caller/target swapped yields the same slug. Verified directly in `test_dm_slug_is_deterministic_min_max_key_sort` by opening the DM as bob first, then as alice, and asserting `bob_result["conversation"] == alice_result["conversation"]`.

## 4. Authorization model

### `tool_comms_kick`
- **Caller key** must validate and resolve to a registered participant.
- **Caller must be a member of the channel** (`conversation in registry.conversations_for(key)`).
- **Caller's per-channel role** (via `RegistryStore.get_channel_role`) must be `'owner'` or `'admin'`. Default-role fallback (no explicit row → `'member'`) means pre-3.0a legacy channels with no owner row reject all kicks until an explicit owner is set, which is the safe semantic (no silent privilege escalation).
- **Target key** must validate, must resolve to a registered participant, AND must be an explicit member of the channel.
- Self-kick is allowed by the contract (an owner can kick themselves out, intentionally — mirrors the privileged-eject semantic; if Phil wants explicit self-kick protection later, it's a one-line add).

### `tool_comms_dm_open`
- **Caller key** must validate + resolve.
- **Target key** must validate + resolve to a registered participant.
- `key == target_key` returns `_error("Cannot open a DM with yourself.")` — self-DM has no sensible two-party semantics.
- No role gate: any registered participant may open a DM with any other registered participant.

The role-side-effect for DMs (both parties → `'owner'`) is symmetric on purpose: there is no admin / member hierarchy inside a two-party DM, and either party should be able to leave / archive / delete without role-table contortion.

## 5. System-message format for kick

Exact body string (asserted byte-for-byte in `test_kick_system_message_published`):

```
[system] {caller_name} kicked {target_name} from #{conversation}
```

Published on `claude-comms/conv/{conversation}/messages` with envelope shape matching `tool_comms_conversation_create`:

```json
{
  "id": "<uuid4>",
  "ts": "<now_iso>",
  "sender": {"key": "00000000", "name": "system", "type": "system"},
  "body": "[system] alice kicked bob from #design",
  "conv": "design",
  "recipients": null,
  "reply_to": null
}
```

**DM-open publishes nothing** by design — a private two-party DM has no audience that needs notification. This is documented in `tool_comms_dm_open`'s docstring.

## 8. [VERIFY] items resolved + new ones surfaced

### Resolved
- **[V-1] DM slug symmetry under sort order.** Confirmed via test 3 (`test_dm_slug_is_deterministic_min_max_key_sort`): opening as bob first then alice second yields the same slug. Python's lexicographic sort on 8-hex strings is deterministic and matches the brief's "sorted alphanumerically" intent.
- **[V-2] DM-create race.** `create_conversation_atomic` returns `None` on `FileExistsError`. The DM tool treats that branch as the idempotent `existed` path rather than as an error, so two simultaneous DM-open calls land cleanly: one creates, the other returns `existed`. Both eventually see the same channel.
- **[V-3] Kick authorization fallback when no store is wired.** When `store=None` (legacy unit-test call sites), the role check is skipped. Production wiring in `mcp_server.py` always passes `_get_registry_store()`, so the gate is enforced on the live path. Test 1 (`test_kick_by_owner_succeeds`) and test 3 (`test_kick_by_non_admin_rejected`) both wire a real store.

### Newly surfaced (NONE blocking 3.5a — flagged for future steps)

- **[NEW-V-A] DM-channel listing in `comms_conversations`.** A DM channel with `visibility="private"` will currently still show up via `tool_comms_conversations` for members (membership read-through). Whether the directory's "Private" tab should split DMs from regular private channels is a frontend question deferred to Wave E.3 Step 3.5b (UI work).
- **[NEW-V-B] DM unread cursors.** DMs use the same `read_cursors` table as any conversation; no special-casing needed. Confirmed via direct read of `RegistryStore.upsert_read_cursor` shape — no schema work required.

For DMs, only the opener can delete even though both parties are 'owner' in the role table. This is a known asymmetry between the meta.created_by field and the role table — flagged for Phil's review in a later step. Not in 3.5a scope.

## 9. Decision log (pragma calls)

1. **Used `registry.leave(target_key, conversation)` (not `registry.remove_membership`).** `ParticipantRegistry` exposes `leave` as the public API; `remove_membership` is `RegistryStore`'s primitive. `registry.leave` already calls through to `self._store.remove_membership` when a store is wired. The brief's wording conflated the two — semantic intent is preserved.
2. **DM-open is silent on MQTT.** No `[system]` message is published when a DM is created. The DM has no audience beyond the two parties, and both will see the channel materialise via their own conversation lists. If Phil wants a "you have a new DM from X" notification on the *target*, that's a Wave E.3 future hook on the auto-join publish path.
3. **DM-open accepts a store but uses it only for role writes.** When `store=None`, the role writes are skipped silently (logged as legacy / unit-test fallback in the docstring). Production wiring always passes a store.
4. **Kick does NOT remove the target's role row.** Only membership is dropped. The role row is preserved so that a re-invite restores the prior role rather than defaulting to 'member'. This mirrors the Slack / Discord semantic and avoids "lost owner on accidental kick" foot-gun.
5. **No retained-clear of presence on kick.** A kicked target's retained presence MQTT message is NOT cleared (unlike `comms_conversation_archive`'s eviction path). Rationale: kick is per-participant and lightweight; presence stale-state self-corrects on the target's next disconnect. If this surfaces as a UX bug, a presence-clear publish is a one-line add.
6. **`conv_data_dir` accepted by both tools but `kick` does not read it.** Kept in the signature for symmetry with the other conv-aware tools and for future use (e.g. if kick ever needs to verify `meta.archived == False`).
