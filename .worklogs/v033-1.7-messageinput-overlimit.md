## 1. Scope

Surfacing the over-limit state for the message composer (UX G-28). Before
this step, when `inputValue.length > MAX_MESSAGE_LENGTH` (10000), the send
button stayed enabled, `sendMessage()` silently early-returned at the
length check, and the user had no idea why their message vanished. This
step makes the over-limit state unmistakable:

- Send button now disabled while over-limit (also while empty-trimmed for
  symmetry with the existing early-return inside `sendMessage()`).
- Inline error banner above the composer in `--ember-400` with a precise
  "N over limit" count and a "Convert to artifact" CTA next to it.
- Textarea wrap border switches to `--ember-400` so the over-limit state
  is visually persistent regardless of focus.
- Character counter color unified to `--ember-400` (was a bare `#ef4444`).
- v0.3.3 stub for the CTA: copies composer contents to clipboard, surfaces
  a transient in-composer notice, AND dispatches a bubbling+composed
  `requestToast` CustomEvent on the composer root so a future App-level
  listener can render a global toast without coupling shape.

The full artifact-create flow (CTA → modal → server-side artifact) is
deferred to v0.4.x and called out as such in the toast text.

### 4c. `convertToArtifact()` v0.3.3 stub

```js
async function convertToArtifact() {
  const text = inputValue;
  const toastText = 'Copied — paste into a new artifact (coming v0.4.x)';
  let copied = false;
  try {
    if (typeof navigator !== 'undefined'
        && navigator.clipboard
        && typeof navigator.clipboard.writeText === 'function') {
      await navigator.clipboard.writeText(text);
      copied = true;
    }
  } catch (err) {
    console.warn('[MessageInput] convertToArtifact: clipboard.writeText failed', err);
  }
  if (!copied) {
    console.info('[MessageInput] convertToArtifact: clipboard unavailable, contents not copied');
  }
  convertNotice = toastText;
  setTimeout(() => {
    if (convertNotice === toastText) convertNotice = '';
  }, 3000);
  if (rootEl && typeof CustomEvent === 'function') {
    rootEl.dispatchEvent(new CustomEvent('requestToast', {
      bubbles: true,
      composed: true,
      detail: { text: toastText, kind: 'info', copied },
    }));
  }
}
```

Key design choices:

- Textarea contents are NOT cleared after the copy. The user may want to
  split the message in-place rather than convert the whole thing — the
  copy is opportunistic, not destructive.
- Clipboard write is best-effort: jsdom doesn't ship a `navigator.
  clipboard` by default, and even real browsers can throw on permissions
  / insecure context. We log + still surface the toast text so the user
  sees confirmation that *something* happened; production deploys with
  `navigator.clipboard` available get the full copy semantics.
- The `requestToast` CustomEvent uses `bubbles: true, composed: true` so
  it walks up from the composer through any shadow boundaries to
  document. Listener shape (`event.detail.text` + `kind` + `copied`)
  documented at the call site.

### 4d. Over-limit textarea border

```css
/*
 * Over-limit textarea border. Wins over the default 1px border on
 * .input-wrap and over the focus-within glow so the over-limit state is
 * visually persistent regardless of focus.
 */
.input-wrap.over-limit {
  border-color: var(--ember-400);
  box-shadow: 0 0 0 1px var(--ember-400), 0 2px 8px rgba(0, 0, 0, 0.15);
}

.input-wrap.over-limit:focus-within {
  border-color: var(--ember-400);
  box-shadow: 0 0 0 2px rgba(245, 158, 11, 0.35), 0 2px 8px rgba(0, 0, 0, 0.15);
}
```

The `:focus-within` rule is doubled up because the base `.input-wrap:focus-
within` rule (line 1380-ish) sets a different `box-shadow` for the
ember-tinted focus glow; without overriding it, the over-limit state
would partially un-render when the user has the textarea focused (which
is most of the time when over-limit, since they just typed the
overflow). Now both states stay consistent.

## 7. Findings surfaced during work

- (S) **`MAX_MESSAGE_LENGTH` is duplicated** between `MessageInput.svelte`
  (`const MAX_MESSAGE_LENGTH = 10000`) and `mqtt-store.svelte.js` (where
  the server-side / cross-cutting limit lives). The brief said to "find

  validation use the same source of truth.
- (S) **`requestToast` CustomEvent has no App-level listener yet**. The
  in-composer `convertNotice` provides immediate user feedback so the UX
  works end-to-end today; the event is forward-compatible scaffolding
  for a future global toast surface. Recommend a small follow-up step to
  add the listener in `App.svelte` once a generic-toast model lands (the
  current App toast list is shaped for incoming-message notifications,
  not generic info text).
- (S) **Over-limit border `:focus-within` override** doubled up the CSS.
  Working as intended (see 4d) but worth flagging if a future visual-
  polish pass decides to redesign the focus glow — the over-limit rule
  needs to keep winning over the base focus-within rule.
- (S) **`disabled || !inputValue.trim()` gating** also disables send for
  empty/whitespace-only composers. This is a small UX bonus that
  previously silently no-op'd at the `sendMessage()` early-return. Not
  in the brief, but it's free and consistent. The new
  `tests/message-input.spec.js` covers this implicitly (the under-limit
  "hello" assertion).

## 9. Outstanding concerns

- **jsdom clipboard fixture**: the new test installs
  `navigator.clipboard` via `Object.defineProperty(...{configurable:
  true}...)` and tears it down in `afterEach`. Some test environments
  make `navigator.clipboard` non-configurable; the `try/catch` around the
  delete handles it. If a future test ordering exposes a leak (clipboard
  mock from one spec affecting another), the centralized cleanup in
  `afterEach` should still fire. Verified with a same-process run of
  `tests/message-input.spec.js` + `tests/toast-improvements.spec.js`
  back-to-back (14 / 14 passing) and the full 26-file suite (421 / 421).
- **CTA only handles the over-limit path**, not "user wants to convert
  any composer contents to an artifact". The brief specifically calls
  this out as a v0.3.3 stub for the over-limit case; the broader artifact-
  conversion affordance is the v0.4.x scope. The current button is gated
  inside `{#if overLimit}` so it only appears when relevant.
- **Vitest delta is +3 not +2**: the third case (`exactly-at-limit`) is
  a boundary safeguard that pins the `>` strict-greater-than semantics
  of the `overLimit` derivation. If a future refactor accidentally
  switches to `>=`, this test catches it before the silent-no-op
  regression resurfaces. Decided to keep all three; brief said "≥ 2".
