## 3. AUDIT FINDINGS

### 3.1 Store-side `unreadHasMention` derivation - GAP FIXED

**Spec point:** "Confirm `unreadHasMention` derivation in the store distinguishes mention messages from ordinary messages."

**What I found:** Only the bootstrap-path (`checkChannels`, mqtt-store.svelte.js:2820-2828) raised `unreadHasMention` from the wire `latest.mentions` list. The live-message dispatch path `#handleChatMessage` (mqtt-store.svelte.js:3683) silently bumped `ch.unread++` without ever inspecting `msg.mentions`. That meant: once you're connected, every mention message lands as an ordinary unread bump - the mention dot only ever surfaces after a reconnect (when bootstrap re-runs). For a chat client where users sit connected for hours, this is the dominant case.

**What now works:** `#handleChatMessage` mirrors the bootstrap-path's mention check. When `msg.mentions` is an explicit array including `userProfile.key`, the channel's `unreadHasMention` flips to true, identical condition to the bootstrap path. Guarded behind the same `channel !== activeChannel` branch as the unread bump so receiving your own active channel doesn't spuriously raise the dot.

**Test that pins the truth:**
- `Step 3.10 - store raises unreadHasMention from live mention messages > mention + muted: live message sets unreadHasMention=true on the channel`
- Five additional tests cover the control cases (ordinary, unmuted, other-user mention, empty list, multi-message sequence).

### 3.2 SidebarChannelRow rendering - CORRECT (already test-locked)

**Spec point:** "Confirm `SidebarChannelRow.svelte` renders the dot variant when `unreadHasMention && muted`."

**What I found:** SidebarChannelRow.svelte:113 derives `showMentionDot = showUnreadBadge && unreadHasMention`, gated only on the section-variant and the unread flag - never on `muted`. The `.muted` modifier on line 161 just reduces row opacity. The mention dot renders correctly when `unreadHasMention && muted`. This was already test-locked by `tests/sidebar-channel-row.spec.js:183` ("mention dot persists when channel is also muted (Design Spec §8.2)"). I added two more tests in the new file for redundancy + to provide a Step 3.10-named landing pad for future regressions.

**Test that pins the truth:** `Step 3.10 - SidebarChannelRow renders mention dot on muted channels > mention + muted: renders row-mention-dot AND .muted class on the row` (and the control: `ordinary + muted: no dot, no numeric badge`).

### 3.3 Toast handler - BUG SURFACED AS [VERIFY] (read-only file)

**Spec point:** "Confirm the toast handler honors 'Only mentions' policy (Step 3.9) by toasting on mention even when muted."

**What I found:** `web/src/App.svelte:400-411` short-circuits the in-app toast on `ch.muted` REGARDLESS of mention status:

```javascript
if (store.inAppToasts && !(ch && ch.muted)) {
  addToast({ ... });
}
```

This violates Design Spec §8.2's invariant for the toast surface: a mention message on a muted channel should still surface a toast (the user explicitly opted into mention-only alerts when they muted). The browser `Notification` API path (`sendNotification` from `web/src/lib/notifications.svelte.js`) is even broader - it always fires for any non-self message regardless of mute, which is the opposite bug (over-notifies on muted ordinary messages).

**Why I did NOT fix it:** App.svelte is on the explicit read-only list in the brief. The fix is one small condition flip (`!(ch && ch.muted) || msgIsMention`), but it sits in App.svelte's `$effect` and changing it without orchestrator approval crosses the wave boundary. Surfaced for Wave G follow-up instead.

**Test that pins the truth:** None in this step (out of scope per file restrictions). See §5 below.
