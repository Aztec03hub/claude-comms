- **New `tool_comms_get_channel_role`** thin wrapper over
  `RegistryStore.get_channel_role` (added by 3.0a). Accepts optional
  `target_participant_key`; defaults to caller's own role.
- **Extended `comms_conversation_update` HTTP wrapper** to mirror the
  new kwargs.
- **New `comms_get_channel_role` HTTP wrapper** following the
  registry-read pattern (`comms_members`).
- **New `_get_registry_store` helper** in `mcp_server.py` (the existing
  `_get_store` returns `MessageStore`, not `RegistryStore`).
- **Reconciled `_serialize_conversation_full` + `get_all_conversations_full`**
  bootstrap-code value sets from the pre-3.6b placeholders
  (`'listed'/'unlisted'` for visibility, `'public'/'private'` for mode)
  to the 3.6b spec-pinned values that ChannelAdminPanel actually emits
  (`'public'/'private'` for visibility, `'open'/'invite'` for mode).
  See §7 for the [VERIFY] item surfaced about the frontend-side filter.

## 5. transferOwnership decision rationale (3.0a [VERIFY] #2 reinterpretation)

3.0a's [VERIFY] #2 cautioned against using `set_channel_role(..., 'member')`
as a downgrade primitive, on the grounds that it "defeats the FK cascade"
when the underlying participant gets deleted. The brief asked me to
re-evaluate for the transferOwnership path.

**Decision**: Use `set_channel_role(old_owner_key, 'member')` to demote
the prior owner during transferOwnership.

**Rationale**: The FK-cascade concern in 3.0a applies to the **participant
deletion** path — if a participant gets deleted and their role row was
explicitly downgraded to `'member'`, that row would survive the cascade
and create stale data. For **transferOwnership**, no participant is
being deleted — the prior owner remains a fully-valid participant who
simply no longer holds the owner role on this one channel. Demoting them
to `'member'` is the correct authorization semantic (anything else would
leave them as `'owner'`, which would mean two owners simultaneously, or
leave them with no explicit row and rely on the default `'member'` — but
the latter loses the audit trail that they USED to be owner).

The 3.0a caution is preserved for its original use case (participant
deletion). Future participant-delete paths should NOT explicitly
downgrade roles before the delete; they should let the FK cascade do
the cleanup. transferOwnership is a separate workflow with separate
semantics.

ChannelAdminPanel's actual emission" directive.

**[VERIFY-3.6b-2]** (frontend follow-up needed, not in this scope):
The frontend `mqtt-store.svelte.js` has TWO surviving references to
the legacy `'listed'` value-set:

- Line 576: `visibility: typeof r.visibility === 'string' ? r.visibility : 'listed'` (rehydration default)
- Line 891: `c.visibility === 'listed'` (available-channels filter)

Now that the backend serves `'public'/'private'`, these need a frontend
Wave C patch to use `'public'` as the rehydration default and `'public'`

This is a frontend issue out of scope per the brief's "ANY .svelte /

Wave C step.

**[VERIFY-3.6b-3]** (mqtt-store frontend admin-action behavior):
The `renameChannel` accessor sends `name` in its payload (verified at
mqtt-store.svelte.js:2354). My extended backend REJECTS that field with
an error envelope (slug is immutable). The frontend's optimistic update
will roll back as expected, so the user-visible behavior is "rename
attempt fails immediately with a clear error". The correct frontend
fix is to switch `renameChannel` to send `display_name` instead. This
is a frontend issue out of scope per the brief; SURFACED for the next
Wave C step.

**[VERIFY-3.6b-4]** (transferOwnership 1-arg form):
ChannelAdminPanel still calls `transferOwnership` with 1 arg (the panel

mqtt-store comment block, lines 2491-2497). The frontend accessor

2-arg path is never exercised by the UI today. The backend is ready
for the picker landing whenever Wave C ships it.

## 8. Decision log (pragmatic calls)

- **Slug-rename guard placement**: Inside the tool, BEFORE the validation
  block on visibility/mode. This guarantees a clear "use display_name"
  error message reaches the frontend even when multiple bad fields are
  passed simultaneously.
- **Single combined system message for multi-field updates**: The
  existing single-field topic-only phrasing is preserved for
  backwards compat with any system-message-parsing consumers. Multi-field
  changes get a new generic `[system] {actor} updated #{conv}: <field
  list>` phrasing.
- **Response shape echoes back the new values**: So the frontend can
  reconcile its optimistic local state without a round-trip
  `comms_read` / `comms_get_channel_role` follow-up. Also includes
  `updated_fields: list[str]` for debugability.
- **transferOwnership only takes effect when `store` is provided**: The
  tool is callable both with and without a `RegistryStore` (tests run
  the registry-only path). When `store` is None, the meta.json side of
  the transfer still lands; the role table is left untouched. The HTTP
  wrapper always passes `_get_registry_store()` so production always
  gets the role swap.
- **noqa pragmas on `tool_comms_conversation_update`**: PLR0912/PLR0913/
  PLR0915 (too many branches/args/statements). The function genuinely
  needs all those branches (one per optional field plus validation +
  transferOwnership side-effects). Splitting would obscure the atomic-
  apply semantic. Accepted.
