## 1. Scope

Three independent cleanups bundled into one focused pass on the high-friction
`mqtt-store.svelte.js` (plus a small touch to `MessageBubble.svelte` for the
G-62 status visualization):

- **Eng R-1**: removed misleading `_len = this.messages.length;`-style
  "dependency-pinning" reads inside every `$derived.by(...)` block in the
  store. Svelte 5's `$state` proxy tracks every synchronous read inside the
  derived expression — including `.filter()` / `.find()` / `Object.keys` /
  `Object.values` — so the `_len`/`_p`/`_ch`/etc placeholders are no-ops at
  best and misleading at worst. The replacement uses plain `$derived(...)`
  for single-expression cases and renames the local aliases to short non-
  underscored names for the few multi-step `$derived.by` blocks that
  legitimately need temp variables. Comment was rewritten to reference the
  `$derived` docs' "Understanding dependencies" section instead of the
  prior mischaracterization.
- **UX G-43**: changed the hardcoded `userProfile.name = 'Phil'` default to
  the sentinel `'(unset)'`, added a `nameUnset = $state(true)` flag, and
  rewrote the `connect()` identity-fetch flow so we adopt the daemon's name
  only when it's a non-empty string; localStorage fallback only fires when
  the daemon hasn't already provided one. New web clients no longer post
  under a random hardcoded identity.
- **UX G-62**: added a per-message `status: 'sending' | 'sent' | 'failed'`
  field to outgoing local-echo messages, a private `#pendingSends` FIFO
  queue (cap 100, drop-oldest on overflow), a `#drainPendingSends`
  helper wired into the `'connect'` callback, a `retryMessage(messageId)`
  public method, and the matching three-state visualization in
  `MessageBubble.svelte` (spinner / nothing / `!` + Retry link).

### 4c. G-62: `#pendingSends` queue declaration + drain skeleton

```js
  /**
   * Outgoing messages queued while we were disconnected from the broker
   * (UX G-62). Each entry is `{ messageId, topic, payload }`. ...
   * @type {Array<{ messageId: string, topic: string, payload: string }>}
   */
  #pendingSends = [];
  /** Hard cap on `#pendingSends` queue length. */
  static #PENDING_SENDS_CAP = 100;
```

```js
  sendMessage(body, replyTo = null, { mentions = null, recipients = null } = {}) {
    ...
    const msg = {
      ...
      // Per-message delivery status (UX G-62). Outgoing local-echo bubbles
      // start as 'sending'. On successful publish ... → 'sent'. On publish
      // error or queue-full eviction → 'failed'.
      status: 'sending',
    };
    ...
    if (this.#client && this.connected) {
      this.#publishOutgoing(msg.id, topic, payload);
    } else {
      // Disconnected — queue instead of silently dropping (UX G-62).
      this.#queuePendingSend(msg.id, topic, payload);
    }
    ...
  }

  #publishOutgoing(messageId, topic, payload) {
    if (!this.#client) {
      this.#updateLocalMessageStatus(messageId, 'failed');
      return;
    }
    try {
      this.#client.publish(topic, payload, { qos: 1 }, (err) => {
        if (err) {
          this.#updateLocalMessageStatus(messageId, 'failed');
        } else {
          this.#updateLocalMessageStatus(messageId, 'sent');
        }
      });
    } catch {
      this.#updateLocalMessageStatus(messageId, 'failed');
    }
  }

  #queuePendingSend(messageId, topic, payload) {
    if (this.#pendingSends.length >= MqttChatStore.#PENDING_SENDS_CAP) {
      const dropped = this.#pendingSends.shift();
      if (dropped) this.#updateLocalMessageStatus(dropped.messageId, 'failed');
    }
    this.#pendingSends.push({ messageId, topic, payload });
  }

  #drainPendingSends() {
    if (this.#pendingSends.length === 0) return;
    const queue = this.#pendingSends;
    this.#pendingSends = [];
    for (const item of queue) {
      this.#publishOutgoing(item.messageId, item.topic, item.payload);
    }
  }
```

## 9. Outstanding concerns

- **Test seams `_installTestClient` / `_drainPendingSendsForTest` /
  `_pendingSendsLengthForTest`** are public methods used only by the
  new vitest spec. They mutate private state (`#client`,
  `this.connected`, the `#pendingSends` array). Marked clearly with
  "Test-only seam" doc comments and `_`-prefixed names; production
  code never calls them. Acceptable given there's no per-file private
  field reflection in JS, but flagged for the orchestrator's awareness.
- **Drain order vs subscribe/history-fetch order**: I put
  `#drainPendingSends()` BEFORE `#fetchHistory()` in the `'connect'`
  callback so the user's queued sends land at the broker first. This
  also means they don't get echoed back via history (history fetch
  runs after). Considered defensible: dedup via `#seenIds` would
  catch the echo if it raced in the other direction anyway, and the
  user mental model — "what I typed offline goes out first" — is
  preserved.
- **`retryMessage` strips local-only fields (`status`, `channel`)
  before re-serializing the payload**: needed because the original
  payload was serialized at compose-time and shipped to `#pendingSends`
  as a JSON string; for retry we re-build from the live message object
  which has `status` mutated on it. The `channel` field is the local
  `#handleChatMessage`-injected mirror, not the wire format. Verified
  by inspecting the wire format in `handleChatMessage` — the broker
  sees `conv`, not `channel`. Spec-correct.
