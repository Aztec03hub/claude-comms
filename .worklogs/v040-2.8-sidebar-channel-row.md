## 4. Design decisions

### 4.1 sectionVariant prop vs context-inference

**Decision:** Caller passes an explicit `sectionVariant: 'starred' |
'active' | 'available'` prop.

**Rationale:** The spec mentions both options ("Caller passes a
`sectionVariant` prop ... OR the component infers from context â€” your
call; document choice in worklog"). I chose the explicit prop because:

1. **Testability:** A test fixture can render a row in any section
   without having to invent a parent context or fake-derive
   membership state from the `channel` fields.
2. **Decouples visual variant from data:** A channel can show up in
   either Active or Starred depending on its `starred` field, BUT
   the visual variant (filled vs hollow star, hover-only vs always-
   visible) is a SECTION concern, not a per-row concern. The Section
   component (Step 2.9) knows which section it is â€” it can pass
   that knowledge in one place rather than every row second-guessing
   its own classification.
3. **Defensive vs the Design Spec Â§2.6 "starred-but-left auto-unstar"
   edge case:** if a channel somehow ends up with `starred=true` in
   the Available section (shouldn't happen with the store
   derivations, but the derivation is in a different file), the row
   still renders the correct Available-style hollow hover-only star.

### 4.2 Row root: `<div role="button">` not `<button>`

**Decision:** Row root is `<div role="button" tabindex="0">` with
keydownâ†’Enter/Space and `aria-pressed={isActive}`. Star icon stays a
real nested `<button>`.

**Rationale:** Svelte autofixer flagged the initial `<button>`-as-row
version: HTML forbids nested `<button>` elements, and the browser
silently repairs the DOM (closing the outer button before the inner
one, breaking Svelte's tree assumptions). The brief said "prefer
`<button>`" but also said the row must contain the star button â€” the
two are incompatible. I picked the div+role version so the star
button stays a real `<button>` (which is what gives it accessible
keyboard focus + Enter activation + the aria-pressed contract for
"starred" state independently from row selection).

The row's keydown handler maps Enter/Space â†’ click, matching the
existing `Sidebar.svelte` v0.3.3 pattern. `aria-pressed={isActive}`
on the row signals selection state to assistive tech.

### 4.3 Member-chip visibility

**Decision:** Hover-only by default (Active + Starred sections);
always-visible for Available section.

**Rationale:** Spec explicitly carves out per-variant chip behavior
(spec line 1366). Active + Starred rows are member channels with
known context; the count is a tertiary detail. Available rows are
new-to-the-user channels where member-count is a primary
"is this active?" signal so it should always show.

### 4.4 Mention dot on muted channels â€” invariant preserved

The mention dot persists on muted rows. The row carries `.muted`
(reduces opacity to 0.5/0.75 on hover), but the dot element renders
regardless of mute state. This is Phil's hard invariant from
Design Spec Â§8.2 and is reasserted in plan line 1022.

Test coverage: `mention dot persists when channel is also muted
(Design Spec Â§8.2)`.

### 4.5 Defensive optional-chaining on the channel prop

Every field-read uses `channel?.<field>`. The store guarantees no
`undefined` leaks in production, but a hand-rolled test fixture (or a
caller that wires up the component before the store finishes
bootstrapping) might omit a field. The defensive read avoids crash-
on-render in those paths â€” same permissive shape the existing
`Sidebar.svelte` pattern uses.
