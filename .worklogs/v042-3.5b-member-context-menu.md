## 3. Right-click hook approach

Per-row `oncontextmenu={(e) => handleContextMenu(e, member)}` listener on
the rendered DOM in MemberList. We chose per-row over event delegation
because:

- The two render paths (online snippet + offline-row block) already emit
  unique elements with `data-testid="member-{key}"`; closure capture of
  `member` is trivial.
- `handleContextMenu` early-returns when `onMemberContextMenu` is
  unwired, so legacy mounts (member-list.spec.js without the new prop)
  continue to surface the browser's native context menu -- no regression
  to existing tests.
- Matches the precedent from `SidebarChannelRow.svelte` (line 171,
  `oncontextmenu={handleContextMenu}`), so the codebase pattern is
  consistent across both sidebars.

The handler calls `event.preventDefault()` only when a parent is wired;
this is a deliberate choice so users who deploy MemberList in
isolation (a future test harness or docs) keep their browser-native
right-click intact.

## 4. Mute localStorage schema

Storage key: `cc:user-muted:{targetKey}`
Value: `'1'` when muted; key absent otherwise (no `'0'`).
Namespace prefix `cc:` chosen to keep this separate from the
`claude-comms.*` namespace used by MemberList's section-expansion
preferences; the new key surface is per-user, not per-UI-component, so
it gets a shorter prefix.

In-memory mirror: `store.userMutes: Record<string, boolean>` (Svelte
`$state` object). Source of truth across page reloads is localStorage;
the in-memory map is the reactive view so components reading
`isUserGloballyMuted(key)` in a `$derived` block re-render on toggles
without polling.

Per Q4 (Phil's locked pattern): this state intentionally does NOT cross
MQTT or MCP. The per-channel mute precedent (v0.4.0) used the same
"personal preference stays client-side" model.

## 8. [VERIFY] items

Resolved during implementation:

- [VERIFY-3.5b-1] Confirmed `comms_dm_open` response payload shape via mcp_tools.py:2698-2790. The `status` field is one of `'opened'` (first create) or `'existed'` (idempotent re-call). Our startDM returns both in the success envelope so callers can branch on "did we just open it for the first time" if they want to render a different welcome message later.
- [VERIFY-3.5b-2] Confirmed `mcpCall` returns `{success, payload, error}` per `web/src/lib/api.js:360`. Both kickMember and startDM read the envelope correctly.

New [VERIFY] items surfaced:

- [VERIFY-3.5b-3] The "kick" affordance is hidden when `currentChannelRole === 'member'` per the role-gate spec, but the backend's `tool_comms_kick` ALSO enforces the role gate at `mcp_tools.py:2664`. There is no client-side visibility test for the role gate other than the menu's render-time decision; a malicious client could bypass the UI hide by calling `store.kickMember` directly from devtools. The backend rejection produces a clean error envelope (test 8 exercises this), so the user-visible behavior on bypass is identical to the legitimate-but-unauthorized path. No regression; spec-conformant.
- [VERIFY-3.5b-4] The `userMutes` map only mirrors localStorage AT WRITE TIME. If a second tab writes a key to localStorage, this tab's `userMutes` won't update reactively until either (a) `isUserGloballyMuted` is called (it falls back to localStorage on map miss) or (b) the tab is reloaded. Per Q4's "personal preference" model this is acceptable; cross-tab sync is a separate optimization. Documented in the JSDoc for `userMutes`.
