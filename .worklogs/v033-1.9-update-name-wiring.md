## 1. Scope

Wired the SettingsPanel display-name input to the daemon's `comms_update_name` MCP tool, closing the UX G-9 showstopper (silent identity divergence). Pre-1.9, `handleNameChange` wrote the new name into `store.userProfile.name` + `localStorage` immediately on every keystroke and never spoke to the daemon, so other connected clients kept seeing the old name forever.

Now: a new `updateName(key, newName)` helper in `web/src/lib/api.js` POSTs JSON-RPC to the FastMCP `/mcp` endpoint (which runs `stateless_http=True` + `json_response=True`, so no session handshake is needed). `SettingsPanel.svelte` debounces input changes by 500ms, blocks the call when `store.connected === false` with the clear copy `Cannot rename while disconnected. Reconnect first.`, shows an `aria-live="polite"` inline status (`Saving...` → `Saved` 1.5s fade, or `Error: <reason>` 3s fade), and only mutates `store.userProfile.name` / clears `store.nameUnset` after the daemon confirms. Failures revert the visible input to the last server-confirmed name.

Six new vitest specs cover the offline gate, the 500ms debounce window (including keystroke coalescing), the success path (store update + nameUnset clear + Saved status), the failure path (input revert + store unchanged + error copy), and the nameUnset hold-on-failure invariant.

### `web/src/lib/api.js` — new `updateName` helper

```diff
@@ end of file @@
+/**
+ * Update the local participant's display name via the daemon's MCP
+ * `comms_update_name` tool (UX G-9 showstopper — wires the SettingsPanel
+ * rename into the protocol layer instead of silently writing only to
+ * localStorage).
+ *
+ * Transport: the daemon's FastMCP server runs with `stateless_http=True`
+ * + `json_response=True`, so a one-shot JSON-RPC `tools/call` POST against
+ * the `/mcp` endpoint succeeds without an initialize handshake. ...
+ */
+export async function updateName(key, newName) {
+  if (!key || typeof key !== 'string') {
+    return { success: false, error: 'Missing participant key.' };
+  }
+  if (!newName || typeof newName !== 'string' || !newName.trim()) {
+    return { success: false, error: 'Name cannot be empty.' };
+  }
+
+  const controller = new AbortController();
+  const timer = setTimeout(() => controller.abort(), 5000);
+
+  try {
+    const res = await fetch(`${API_BASE}/mcp`, {
+      method: 'POST',
+      headers: {
+        'Content-Type': 'application/json',
+        Accept: 'application/json, text/event-stream',
+      },
+      body: JSON.stringify({
+        jsonrpc: '2.0',
+        id: 1,
+        method: 'tools/call',
+        params: {
+          name: 'comms_update_name',
+          arguments: { key, new_name: newName },
+        },
+      }),
+      signal: controller.signal,
+    });
+
+    if (!res.ok) return { success: false, error: `Server returned HTTP ${res.status}.` };
+    const body = await res.json();
+    if (body && body.error) return { success: false, error: body.error.message || 'Server rejected the rename.' };
+
+    const result = body?.result || {};
+    let payload = result.structuredContent;
+    if (!payload && Array.isArray(result.content)) {
+      const textBlock = result.content.find((c) => c && c.type === 'text');
+      if (textBlock && typeof textBlock.text === 'string') {
+        try { payload = JSON.parse(textBlock.text); } catch { payload = null; }
+      }
+    }
+
+    if (!payload) return { success: false, error: 'Server returned an empty response.' };
+    if (payload.status === 'error') return { success: false, error: payload.error || 'Rename failed.' };
+    if (payload.status === 'updated' && payload.name) return { success: true, name: payload.name, key: payload.key };
+    return { success: false, error: 'Unexpected server response.' };
+  } catch (err) {
+    if (err && err.name === 'AbortError') return { success: false, error: 'Request timed out.' };
+    return { success: false, error: (err && err.message) || 'Network error.' };
+  } finally {
+    clearTimeout(timer);
+  }
+}
```

## 6. Tests added / modified

- `web/tests/settings-panel.spec.js`: NEW, 6 specs:
  1. `offline gate > blocks the backend call when store.connected is false` — verifies the disconnected hint surfaces immediately, `updateName` is never invoked, and `store.userProfile.name` stays at its original value.
  2. `debounce window > only calls updateName after the 500ms debounce expires` — fires an input event, asserts no call at 400ms, asserts a single call at 550ms.
  3. `debounce window > coalesces rapid keystrokes into a single backend call` — types 4 characters at 100ms intervals (each resets the debounce), confirms 0 calls during the burst and exactly 1 call after the 500ms idle window.
  4. `successful rename > updates store.userProfile.name and clears nameUnset on success` — mocks a `{success: true, name: 'Phil'}` response, asserts both `store.userProfile.name` and `store.nameUnset` flip, and the inline status reads `Saved`.
  5. `failed rename > reverts the input and surfaces the error message on failure` — mocks a `{success: false, error: 'Invalid name "bad/name".'}` response, asserts the input DOM value reverts to the original name, the store is unchanged, and the inline status reads `Error: Invalid name "bad/name".`.
  6. `failed rename > does not flip nameUnset when the server rejects the rename` — invariant lock that a failed rename leaves `store.nameUnset` true so the name-unset banner persists until a real rename lands.

## 7. Findings surfaced during work

- **(S) `Notification` API not available in JSDOM** — `SettingsPanel.svelte:32` reads `Notification?.permission` at component-init time, but JSDOM does not ship the Notifications API, so every render in tests throws `ReferenceError: Notification is not defined`. Worked around by stubbing `globalThis.Notification` at the top of `tests/settings-panel.spec.js`. There is no existing project-wide `vitest setup` file (vitest.config.js has `setupFiles: []`), so a global stub would have been heavier-weight than a 7-line top-of-file shim. Future tests that mount SettingsPanel will need the same stub. Could be folded into a future shared test-setup file when a second spec needs it.
- **(S) `value={displayName}` vs `bind:value={displayName}`** — the current template uses an unbinding attribute, which Svelte 5 still re-renders correctly when `displayName` reassigns from JS (verified by spec 5 — `input.value === 'Original'` after the failure-branch revert). The legacy code that I replaced used the same pattern, so I left it. Switching to `bind:value` would be a stylistic refactor with no behavior change.
- **(S) `vi.runAllTimersAsync` flushes past the fade timers** — initial test runs failed because flushing all timers expired the SAVED_FADE_MS (1500ms) / ERROR_FADE_MS (3000ms) windows before the assertion could observe the post-success / post-failure status. Fixed by advancing exactly through the 500ms debounce + yielding 2 microtasks (`await Promise.resolve()` twice) for the mocked promise resolution, then `await tick()`. This is a vitest pattern future tests of the same status-transition shape should reuse.

## 9. Outstanding concerns

**1. Daemon-side REST shortcut deferred to v0.4.0.** Per the Step 1.9 brief: "If there's NO clean REST path and adding a daemon-side `/api/identity/name` endpoint would simplify the call, NOTE THIS in worklog §9 and use the MCP path anyway as a v0.3.3 stub. The daemon-side endpoint can be added in v0.4.0." This is exactly the situation.

The cleanest long-term path is a dedicated `POST /api/identity/name { key, name }` route added to `src/claude_comms/cli.py` alongside the existing `GET /api/identity` handler (around line 908-918). That would:

- Eliminate the JSON-RPC wrapper boilerplate in `api.js::updateName` (drop `~30 lines` of result-envelope unwrapping for `structuredContent` / `content[0].text`).
- Give the route a stable shape independent of FastMCP transport-version churn (FastMCP could one day require a session handshake even in stateless mode).
- Let the route inherit the bearer-token gating that `POST /api/artifacts/...` already uses, raising the bar against a rogue browser tab on the same machine renaming the daemon's identity without consent.
- Mirror the structure of the rest of the `/api/...` surface, instead of being the lone consumer of `/mcp` from the browser.

For v0.3.3, the JSON-RPC stub is sufficient: the daemon's `stateless_http=True` + `json_response=True` config makes one-shot POSTs work without session state, and the response unwrapping in the helper handles both `structuredContent` and the older `content[0].text` text-block format. The user-visible behavior (rename actually propagates, other clients see the new name) is identical.

**2. No bearer token attached to the rename request.** The MCP layer authenticates by matching the `key` argument against the registry — same surface a TUI / CLI / MCP client would use, no different. The bearer token in `apiPost` is for the `/api/artifacts/*` write surface, which is a separate concern. If we later move rename behind a REST endpoint (per concern #1), we should attach the token at the same time.

**3. End-to-end smoke (real broker) not attempted by this agent.** The Step 1.9 brief notes: "Orchestrator post-handoff: read worklog, verify, then verify end-to-end with a real broker if possible (start daemon, open web UI, change name, watch MQTT presence republish via mosquitto_sub)." That smoke is left for the orchestrator. The vitest specs cover the wiring at the unit level; the daemon-side `comms_update_name` was already verified by pre-existing pytest specs (1207 passing unchanged).
