## 5. Implementation notes

### Focus context rule

Every binding except Escape is suppressed when the keydown target is an
INPUT, TEXTAREA, SELECT, or contenteditable element. `preventDefault()`
is also skipped on the suppressed path so the user's typing isn't lost.

### Symbol-key Shift handling

US keyboard layout types `?` as Shift+/. The browser fires
`{key:'?', shiftKey:true}`. Naïve serialization would yield `Shift+?`,
which would never match a `?` registration. `serializeCombo` solves this
by including Shift in the canonical combo ONLY when the key is a letter
or digit (so `Ctrl+L` vs `Ctrl+Shift+L` still differ) or a multi-char
key (Escape, ArrowUp, ...). For symbols, Shift is the typing mechanism,
not a distinct modifier.

### Ctrl+W browser hijack fallback

Most browsers intercept Ctrl+W to close the tab and can't be overridden.
The registry's `preventDefault()` runs first and might prevent the
hijack in some contexts, but users on browsers where it fails can use
the documented `Ctrl+Shift+W` alias — both bindings call
`store.leaveChannel(active)`.

### Sidebar wiring

The Sidebar's `onBrowseChannels` prop now opens the new
`ChannelDirectoryModal` (`showChannelDirectory = true`) instead of the
legacy `ConversationBrowser`. The legacy component is still mounted
(defensive — could still be set programmatically) but no UI surface
toggles it.

## 7. Risks / known limitations

- Ctrl+W in Chrome / Edge: the browser usually intercepts BEFORE
  application JavaScript. The fallback `Ctrl+Shift+W` is documented in
  the help overlay for users who hit this.
- Help overlay rendering order: `Object.entries(keyboard.descriptions)`
  yields keys in insertion order, which matches the registration order
  in App.svelte's effect. If a future caller registers bindings in a
  different order, the help-overlay row order will shift. Tests don't
  assert order today; this is by design.
