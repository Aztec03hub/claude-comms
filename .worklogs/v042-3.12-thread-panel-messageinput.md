## What shipped

The pre-3.12 ThreadPanel composer was a stripped-down inline
`<input type="text">` + send button with none of the polish that
`MessageInput.svelte` ships in the main channel pane:

- no `@mention` autocomplete
- no slash-command registry (`/me`, `/help`, `/who`, `/clear`, ...)
- no over-limit handling (G-28)
- no format toolbar
- no snippet inserter
- no emoji-picker trigger
- no typing-indicator surface

Step 3.12 swaps the inline composer for a full `MessageInput` mount
routed through a **thread-scoped store proxy** that stamps
`parentMessage.id` as `replyTo` on every default-path send. All of the
above features now work inside threads with zero changes to
`MessageInput.svelte` (read-only per the wave's scope rules).

## MessageInput integration approach

MessageInput's $props are `{ store, channelName, typingUsers, onOpenEmoji }`.
None of them mention threads. To stay scope-clean (no MessageInput
modification), ThreadPanel constructs a `threadStore` $derived object:

```js
let threadStore = $derived(
  useSharedComposer
    ? {
        get participants() { return store.participants; },
        get userProfile() { return store.userProfile; },
        get activeChannel() { return store.activeChannel; },
        get activeMembers() { return store.activeMembers; },
        get channelsById() { return store.channelsById; },
        get composerPrefill() { return store.composerPrefill; },
        set composerPrefill(v) { store.composerPrefill = v; },
        notifyTyping: (...args) => store.notifyTyping?.(...args),
        sendMessage: (body, replyTo = null, opts = {}) => {
          const effectiveReplyTo = replyTo ?? parentMessage.id;
          return store.sendMessage(body, effectiveReplyTo, opts);
        },
        joinChannel: (...args) => store.joinChannel?.(...args),
        leaveChannel: (...args) => store.leaveChannel?.(...args),
        closeChannel: (...args) => store.closeChannel?.(...args),
        setTopic: (...args) => store.setTopic?.(...args),
        setStar: (...args) => store.setStar?.(...args),
        setMute: (...args) => store.setMute?.(...args),
      }
    : null,
);
```

Property getters delegate each read to the underlying store so
MessageInput's `$derived` / `$effect` chains keep observing the same
reactive `$state` proxies — reactivity is not lost by the indirection.
`composerPrefill` needs a setter too because MessageInput writes back
to clear the prefill after the effect fires.

Slash-command registry methods (`joinChannel`, `leaveChannel`,
`setTopic`, etc.) are forwarded so slash commands fired from inside a
thread act on the underlying channel — they're not no-ops.

## Slash-command-in-thread verification

Test 4 (`/me`) and test 5 (`/help`) directly exercise the slash registry
inside the thread composer. Both pass. `/me` proves the
`sendAs` envelope path also gets its `replyTo` rewritten;
`/help` proves the side-effect-only path is unaffected (no phantom
sendMessage). The other ten registered commands aren't pinned by
this step but inherit the same behavior since they all flow through
the same registry.

## [VERIFY] items for the orchestrator

1. **App.svelte mount unchanged.** Per the wave's read-only list,
   `web/src/App.svelte` was NOT updated to pass `store={store}` instead
   of `onSendReply={...}`. Result: the production thread mount still
   uses the legacy inline composer; the new MessageInput composer is
   only exercised by tests. To realize the UX G-36 benefit for end
   users, a follow-up step needs to swap the App.svelte call site to:

   ```svelte
   <ThreadPanel
     parentMessage={threadParent}
     messages={...}
     onClose={() => { showThreadPanel = false; threadParent = null; }}
     {store}
     channelName={store.activeChannel}
     typingUsers={store.activeTypingUsers}
     onOpenEmoji={() => showEmojiPicker = !showEmojiPicker}
   />
   ```

   No new ThreadPanel props are needed for the swap — this is purely
   a call-site change. Suggested follow-up step: `3.12b` (or fold
   into the v0.4.2 release prep step). Once landed, the
   `{:else}` legacy branch in ThreadPanel can be deleted along with the
   `onSendReply` prop.

2. **Reactivity of the proxy under runtime store re-assignment.** The
   `$derived` rebuilds the proxy whenever `useSharedComposer` flips,
   which happens only on mount/unmount transitions in practice
   (store doesn't get hot-swapped at runtime). The test suite mounts
   the panel with the store already in place; we don't exercise the
   "store hot-swapped under the panel" edge case. If a later refactor
   introduces such a path, add a regression test.

3. **`participants` / `currentUser` / `Avatar` import unused.** These
   were unused in the pre-3.12 file too — the `Avatar` import is
   dead, and `participants` / `currentUser` props are received but
   never read (the avatar inside the panel uses
   `parentMessage.sender.name.slice(0, 2)` instead). Left in place to
   minimize the refactor diff; the autofixer doesn't flag them.

4. **Test count.** vitest baseline on this worktree is 745, not the
   "901" called out by the parent orchestrator message. After Step
   3.12: 752 (745 + 7). The parent's "907" target may include other
   suites (pytest, e2e); within vitest the structural requirement
   "≥ 6 new tests" is met (we shipped 7).
