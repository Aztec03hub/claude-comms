## 1. What shipped

- New `ChannelRole` Literal in `participant.py` (`owner` / `admin` / `member`) plus `DEFAULT_ROLE` (`"member"`) and `OWNER_ROLE` (`"owner"`) constants.
- New `conversation_roles` table in `RegistryStore` (composite PK on `(conversation, participant_key)`, CHECK constraint on role, FK CASCADE on `participants.key`).
- `SCHEMA_VERSION` bumped 1 -> 2 with a one-shot creator-grandfather backfill that maps `meta.created_by` (display name) to a participant key via the pre-existing `LOWER(name)` index, seeding `'owner'` per channel.
- Three new public methods on `RegistryStore`:
  - `get_channel_role(conversation, participant_key) -> ChannelRole` (defaults to `"member"`)
  - `set_channel_role(conversation, participant_key, role) -> None` (upsert, refreshes `granted_at`)
  - `list_channel_roles(conversation) -> dict[str, ChannelRole]` (explicit rows only)

## 3. Backfill behavior verified

The 1->2 migration runs `_backfill_creator_roles_v1_to_v2` once, gated on `prior_version == 1` so fresh installs and v2-and-later restarts skip it. For each conversation meta found on disk:

- Empty / `system*` creators are skipped (reserved labels - no real participant key maps to them). Verified via `test_migration_smoke_v1_db_opens_clean_at_v2` which seeds a `general` channel with `created_by='system'` and asserts no role row is inserted.
- Exactly-one match: `INSERT OR IGNORE` seeds `(conv_name, key, 'owner', now)`. Verified via `test_backfill_grandfathers_creator_as_owner` (participant `claude` key `1234abcd`, channel `general` -> `get_channel_role('general', '1234abcd') == 'owner'`).
- Ambiguous match (more than one participant by name): logs a WARNING containing the word "ambiguous", inserts no row, and leaves the channel un-owned until an explicit `set_channel_role` call. Verified via `test_backfill_skips_ambiguous_name_collision` with caplog assertion.
- Zero matches (creator name has no corresponding participant): logs at INFO and skips. Covered by the same ambiguous test branch's negative assertions.

`INSERT OR IGNORE` ensures any pre-existing role row from a manual repair is never clobbered.

## 5. Migration tested on populated v0.4.0-era registry.db fixture

`_seed_v1_db(tmp_path)` helper opens a raw `sqlite3` connection, applies only the v1 portion of `_SCHEMA_DDL` (split on the `conversation_roles` boundary so the fixture cannot drift from production schema), and pins `schema_meta.schema_version = '1'`. Tests using this fixture:

- Pre-assert: `schema_version == '1'` and no `conversation_roles` table.
- Seed via raw SQL: insert participant `claude` (key `1234abcd`) and create `general/meta.json` with `created_by='claude'`.
- Open via `RegistryStore.open(data_dir)` which fires `_init_schema` -> backfill.
- Post-assert: `schema_version == '2'`, `get_channel_role('general', '1234abcd') == 'owner'`, `list_channel_roles('general') == {'1234abcd': 'owner'}`.

All assertions hold. No errors raised during the migration on a populated v1 database.

## 6. [VERIFY] items resolved + new ones surfaced

Resolved from spec:
- [VERIFY] `created_by` stores a display name not a participant key: confirmed via `conversation.py` line 41 (`description="Participant name, or 'system' / 'system-backfill' / 'system-implicit'"`). Backfill uses `LOWER(name)` join accordingly.
- [VERIFY] Ambiguous display-name collision: defensive handling implemented as specified (WARNING + skip, no destructive write).

New [VERIFY] items surfaced for downstream work:
- [VERIFY Wave A Step 3.1 frontend]: the frontend should treat `get_channel_role`'s default `'member'` response as authoritative for legacy data - do not retry-promote based on `created_by` heuristics, since that contract has now moved to the explicit role row.
- [VERIFY Wave A2 Step 3.14]: when `mcp_tools.py` adds role mutation tools, it must NOT independently call `set_channel_role(... 'member')` to "downgrade" - the explicit `'member'` row blocks the FK CASCADE from cleaning up after a participant delete + re-add cycle (the row would survive). Recommend exposing a `clear_channel_role` admin path instead, as foreshadowed in spec test 14's docstring.
- [VERIFY follow-up for the orchestrator]: there is no UI today to set `'member'` explicitly. Step 3.5 may want to design the admin UI so the default state is "no row" rather than "explicit member row" to keep the table small and the cascade semantics intuitive.

## 7. Decision log (pragma calls not in the spec)

1. **`_init_schema` prior-version probe wrapped in `try/except sqlite3.OperationalError`**: spec assumes `schema_meta` exists; on a brand-new DB it does not yet. Wrapped the `SELECT` so a fresh install does not crash, then re-pins the version after migrations succeed. Avoids a hard regression on first-boot installs (no `registry.db` yet).
2. **Threaded `data_dir` into `RegistryStore.__init__`** as an optional kwarg defaulting to `db_path.parent`. The backfill needs `data_dir` to call `list_all_conversations`. Default preserves backward compat with any external caller constructing `RegistryStore(db_path)` directly.
3. **Skipped backfill for `created_by` starting with `'system'`** (covers `'system'`, `'system-backfill'`, `'system-implicit'`). These are reserved labels and have no participant key by design - querying them would always log "unknown creator" noise.
4. **Lazy import of `conversation` inside `_backfill_creator_roles_v1_to_v2`** to avoid pulling `conversation.py` into module-level import. Defensive against any future circular dependency if `conversation` ever needs registry types.
5. **Forward-version WARNING**: if a downgraded daemon opens a DB at a higher `schema_version` than `SCHEMA_VERSION`, log a WARNING rather than refuse to open. Better to keep the daemon up than silently corrupt newer-schema rows the operator may be trying to recover.
