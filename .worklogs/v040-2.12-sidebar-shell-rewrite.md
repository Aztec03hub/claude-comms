## 1. Scope

Replaced the monolithic v0.3.x `Sidebar.svelte` (716 lines, ad-hoc row
markup, hardcoded section gates, inline starred/conversations divide,
hand-rolled search filter) with a thin shell (295 lines) that:

1. Renders three `SidebarChannelSection` instances — Starred / Active /
   Available — each fed by the matching `store.{starred,active,available}Channels`
   pre-sorted `$derived` projection.
2. Houses the right-click `ChannelContextMenu` mount + state
   (`contextMenuOpen`, `contextMenuChannel`, `contextMenuEvent`) plus
   the action router (`handleContextAction`) that dispatches each
   context-menu `actionId` to the appropriate store method.
3. Houses the `LeaveChannelDialog` mount + state plus the pre-leave
   gate (`shouldConfirmLeave`) that mounts the heavy-leave confirmation
   modal when any of (channel.starred, my-authored pinned message,
   >50 my-messages) holds; otherwise fires `store.leaveChannel`
   silently.
4. Owns the brand header (logo + version label) and the search input
   (still UI-only — Step 2.14's keyboard-shortcut work will hook
   Ctrl/Cmd+K here).
5. Owns the footer chrome — user avatar, name, three-state connection
   status pip + label, settings cog — preserving v0.3.3 G-5 + G-25
   testids and behavior.

SORT-LOCK invariant: 0 `.sort(` calls in `Sidebar.svelte`. Sections
receive already-alphabetized arrays from the store and render rows in
delivered order.

## 5. Approach + key decisions

- **Pre-leave gate computed on demand**: the brief's pseudocode used
  `store.messageCounts?.[c.id]` and `store.pinnedMessages?.[c.id]?.some(...)`
  shapes that don't match the actual store. `pinnedMessages` is a flat
  array with `channel` + `from` fields per entry; there is no
  `messageCounts` map. I implemented `countMyMessages(id)` and
  `hasMyPinnedMessages(id)` as local helpers that filter the
  appropriate flat arrays. Cost is O(messages) per leave click, which
  is acceptable because leave is a rare interaction.

- **isCreator check uses `userProfile.key`, not name**: the brief used
  `createdBy === store.userProfile?.name`, but the store consistently
  stores creator identity by `key`. Using `name` would have produced
  false negatives whenever the user's display name differed from the
  key (which is the common case — keys are stable identifiers, names
  are mutable). Wired isCreator against `userProfile.key` to match
  the store's contract.

- **Star toggle: prop > store.setStar fallback**: the brief specified
  `store.setStar(id, !c.starred)` but the existing `onStarToggle` prop
  drilling pattern (v0.3.3 Step 1.5b) expects the parent to be able to
  intercept star toggles. Preserved that: `handleStarToggle` first
  checks `typeof onStarToggle === 'function'` and only falls through
  to `store.setStar` when no prop is wired. This keeps
  `prop-drilling.spec.js` Sidebar suite passing AND matches what App.svelte
  has been doing since v0.3.3.

- **Available-section row click = join + switch**: clicking a row in
  the Available section calls `store.joinChannel(id)` and, on success,
  follows up with `store.switchChannel(id)` so the user lands in the
  channel they just joined. The brief said "Available calls
  `store.joinChannel(id)`" but didn't specify the post-join switch —
  this is the obvious UX (the user clicked the channel because they
  want to read it).

- **Delete action: window.confirm placeholder**: the type-name-to-confirm
  modal is a Step 2.13 deliverable. For v0.4.0 we ship a
  `window.confirm` fallback so creators can still delete; the
  contextual menu still routes the action correctly.

- **`onMuteChannel` prop dropped**: the legacy prop is now an
  orphaned channel. The context menu's `mute:all` / `mute:mentions` /
  `mute:off` actions call `store.setMute` directly. Cleaned up both
  Sidebar.svelte's `$props()` destructure AND App.svelte's invocation.

- **CSS preserved + compacted**: the brand, search-input, footer,
  status-dot, and create/browse buttons all live in Sidebar.svelte
  and need their styles. I compacted multi-line CSS rule blocks into
  single-line declarations to hit the file-size budget (< 300 lines).
  Visual output is unchanged — only whitespace changed in CSS.

## 9. Flagged for orchestrator follow-up

- **Search input is UI-only.** The brief left filtering out of scope
  for Step 2.12; Step 2.14 (sidebar shortcuts) is the natural home
  for a Cmd+K command palette that hooks the input. Documenting here
  so the search input isn't accidentally torn out for being "dead".

- **Mark-all-read context-menu action is a no-op for v0.4.0.** The
  store does not currently expose a `markAllRead(id)` method — Phil's
  v0.4.1 backlog has it scheduled. The handler accepts the
  `mark-read` actionId and silently dismisses the menu so the user
  experience is at least non-broken; v0.4.1 will fill in the actual
  read-cursor write.

- **Info action falls through to onBrowseChannels.** Step 2.13's
  `ChannelDirectoryModal` will accept a "scroll to row X" instruction;
  until then, the menu's "Channel info" item just opens the directory.

- **Delete uses window.confirm.** Type-name confirmation is a
  Step 2.13 / v0.4.1 deliverable. Documented inline in
  `handleContextAction`.

- **15-second undo toast for silent leave is NOT wired here.** The
  store's `leaveChannel` returns `{ done, cancel }` but Sidebar.svelte
  doesn't currently hook the App.svelte toast system. That wiring
  belongs in App.svelte (Step 2.14 or as a v0.4.0 polish ticket);
  for v0.4.0 the leave still goes through the store's
  `#scheduleUndoable` wrapper which delays the actual MCP commit by
  15 seconds and would honor an external `cancel()` call — so no
  data is at risk, only the visual undo affordance is missing.
