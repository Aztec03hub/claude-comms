# v0.4.2 Step 3.14 — Profile status backend (Wave A2 re-issue post-§I.18-collision-rename)

## 1. WHAT shipped

- New durable per-participant profile_status triplet (emoji / text / expires_at) persisted on the `participants` SQLite row.
- Two new MCP tools: `comms_profile_status_set(emoji, text, expires_at=None)` and `comms_profile_status_clear()`. Caller identity derived single-tenant from `config.identity.key`.
- MQTT presence payload on the existing retained topic `claude-comms/presence/{key}/{connKey}` augmented with the three `profile_status_*` keys; fan-out per active connection.
- Schema migration `SCHEMA_VERSION` 2 → 3 with idempotent ALTER TABLE block gated by `PRAGMA table_info(participants)`.
- Standalone `auto_expire_profile_statuses_loop` coroutine (~60s tick) attached at daemon startup; sweeps expired triplets and republishes the cleared payload retained.
- The v0.4.0 ephemeral activity API (`comms_status_set` / `comms_status_clear`) remains untouched and continues to pass its full suite.

## 5. Auto-expire coroutine: piggyback vs standalone

**Decision: standalone** — added a new `asyncio.create_task(auto_expire_profile_statuses_loop(...))` in `cli.py` at daemon startup.

**Rationale**:

- The brief explicitly marks `src/claude_comms/presence.py` (the `PresenceManager` owner file) as read-only for this step.
- Piggybacking would require inlining a call inside `PresenceManager._sweep_once` or attaching a callback hook — neither is possible without modifying `presence.py`.
- Estimated piggyback LOC if presence.py were writable: 8 LOC (one new optional callback parameter on `PresenceManager.__init__`, one block in `_sweep_once`, one wiring change in `cli.py`). Standalone path is 23 LOC total (loop + wiring) and zero-impact on `presence.py`.
- The brief's §[VERIFY] previous-#2 explicitly authorised this fallback: "If the integration would require restructuring PresenceManager (>20 LOC of refactor in someone else's code), keep it standalone."

The standalone path also keeps the sweep cadence independent (60s vs the presence default 30s), which is the brief's pinned tick.

## 9. Decision log

- **Two-stage publish helper (`_build_profile_status_presence_payload` + `_publish_profile_status_to_all_connections`)** instead of inline publish in the tool — improves testability (presence payload smoke can call the builder directly) and reuses the helper from the auto-expire coroutine.
- **In-memory `Participant` mirror updated in-tool** so subsequent `/api/participants` GETs and `comms_members` calls see fresh values without re-reading the registry.db. Alternative was a registry-method `reload_participant` round-trip; the in-place mutation is consistent with how the existing `Activity` field is handled on `ConnectionInfo`.
- **Standalone auto-expire** (per §5) because the brief made `presence.py` read-only.
- **`expires_at` stored as raw ISO string in SQLite**, parsed into `datetime` only for the in-memory `Participant.profile_status_expires_at` field. This matches how the existing `Activity.expires_at` field stores ISO strings (no datetime serde) and lets the auto-expire sweep do a simple string comparison via the existing `now_iso()` helper.
- **Both-None collapses to clear** (§6 test) — chosen because the alternative (write three NULLs and return `status="set"`) is semantically indistinguishable from clear and creates an ambiguous API surface.
- **`_do_publish` widened in `cli.py`** instead of wrapping at the call-site, because the latent bug already affected `publish_mcp_presence_on_join` / `_publish_offline`. The widening is a strictly compatible default-False extension.
