## 1. Scope

New confirmation modal for leaving a channel where the user has invested
enough in it that an accidental leave would sting. Per Design Spec §4.1
and architecture spec Step 2.11. The parent (Sidebar in Step 2.12)
decides whether to mount this dialog based on three trigger conditions:
- `messageCount > 50` (user has sent enough messages to care), OR
- `hasPinnedMessages === true` (user authored pinned content here), OR
- `isStarred === true` (auto-unstar warning per Spec §2.6).

If none of those hold, the parent skips the dialog and the leave proceeds
silently. This component does NOT gate; it renders and reports user
intent. Trigger logic lives in Step 2.12.

Six props per the brief:

```svelte
let { channel, messageCount, isStarred, hasPinnedMessages, onConfirm, onCancel } = $props();
```

## 4. Design decisions

### 4a. Plain DOM, not bits-ui Dialog.Portal

The existing `ConfirmDialog.svelte` uses `bits-ui` Dialog primitive,
which renders into a `Dialog.Portal` outside the component subtree.
That's the right call for production a11y, but it makes JSDOM tests
brittle: `render()` returns a container that does NOT contain the
portal'd content, so queries via `getByTestId` miss the dialog
entirely. There are NO existing tests for ConfirmDialog or ChannelModal
in this repo — they were both shipped pre-test-discipline.

Since the brief explicitly demands ≥ 6 new tests covering focus trap,
escape/enter behavior, and default-focus-on-Cancel, I chose to render
plain DOM (overlay + dialog divs) and own the a11y machinery
explicitly:
- `role="dialog"` + `aria-modal="true"` + `aria-labelledby`
- Self-implemented focus trap (Tab / Shift+Tab cycle between the two
  buttons)
- Self-implemented Escape handler on the dialog root
- `$effect` capturing `document.activeElement` pre-mount and restoring
  on unmount
- Default focus via `queueMicrotask` so the binding is live

This trades the bits-ui FocusScope for ~30 lines of explicit a11y
code, but makes every a11y guarantee directly testable and removes
the JSDOM-portal escape hatch. Phil's pattern of "explicit
interactions, not magic" applies.

### 4b. Default focus on Cancel, not Leave

Per the brief: "Destructive default-focus is the user actively picking
to proceed." Two pieces:
- `$effect` queues `cancelBtn?.focus()` inside `queueMicrotask` so it
  runs after Svelte's mount phase wires the `bind:this`.
- The brief is explicit that global `Enter` should NOT confirm. Only
  Enter pressed while the Leave button has focus confirms. So a user
  who lands on Cancel and hits Enter activates Cancel (browser default
  button activation) → `onCancel`, NOT `onConfirm`.

This is verified by the `'Enter on the Cancel button does NOT call
onConfirm'` spec (line ~190 in the spec file).

### 4c. Focus trap implementation

Implemented at the dialog root via `onkeydown={handleDialogKeydown}`.
The handler queries focusable elements at keydown-time, so adding a
third focusable later (e.g. a "Don't ask again" checkbox in a future
spec change) requires NO code change here.

The trap cycles:
- Tab on last focusable → first focusable
- Shift+Tab on first focusable → last focusable
- Tab from outside the dialog (e.g. if focus somehow escaped) → snap
  back to first focusable

### 4d. Focus restoration

Captured in the same `$effect`'s synchronous body BEFORE
`queueMicrotask` fires the default-focus shift. So `previouslyFocused`
holds the pre-mount `document.activeElement`. The cleanup function
restores it on unmount.

Verified by the `'restores focus to the element that was focused
before mount, on unmount'` spec.

### 4e. Overlay click cancels; content click does not

Plain DOM gives us:
- Overlay div has `onclick={handleOverlayClick}` → calls `onCancel`.
- Dialog div has `onclick={handleContentClick}` which calls
  `e.stopPropagation()` so clicks inside the dialog body don't bubble
  up to the overlay handler.

Verified by the `'clicking inside the dialog content does NOT bubble
to overlay cancel'` spec.

### 4f. Singular/plural copy

Required for grammatical correctness: `'1 message'` vs `'2 messages'`.
Verified by two separate specs.

## 5. Code structure

```
<div class="leave-overlay" onclick={cancel} role="presentation">
  <div bind:this={dialogEl} role="dialog" aria-modal aria-labelledby onkeydown={trap}>
    <h2 id={titleId}>Leave #{channel.name}?</h2>
    <body id={descId}>
      <p>You've sent {N} message(s) ...</p>
      {#if isStarred} <warning/> {/if}
      {#if hasPinnedMessages} <warning/> {/if}
    </body>
    <footer>
      <button bind:this={cancelBtn} onclick={cancel}>Cancel</button>
      <button onclick={confirm} onkeydown={enter→confirm}>Leave channel</button>
    </footer>
  </div>
</div>
```

Bindings:
- `cancelBtn` — for default-focus via `cancelBtn.focus()` in `$effect`.
- `dialogEl` — for the focus trap to scope its `querySelectorAll`.

The Leave button does NOT need a `bind:this` because its Enter handler
is local (`onkeydown={handleConfirmKeydown}`) and click activation is
native.
