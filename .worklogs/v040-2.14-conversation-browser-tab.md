## 1. Scope

Refactor the single-mode `ConversationBrowser.svelte` (a v0.3.x slide-out panel
that called `/api/conversations?all=true` and ran its own local sort) into a
DUAL-MODE component:

* **Standalone mode** (default, back-compat for the existing
  `App.svelte` call site at line 549-558): renders the outer panel chrome
  (header bar + close button + slide-in container) + an INTERNAL filter
  input + the row body content. Behaves like v0.3.x but with the new
  sub-section partitioning underneath.
* **Embedded mode** (`embedded={true}`): renders ONLY the body content.
  The parent `ChannelDirectoryModal.svelte` (Step 2.13, sibling worktree)
  provides the modal chrome, the filter input, and the locked sort dropdown.
  The component consumes the parent's filter via the new `filterValue`
  prop, the parent's sort via `sortKey`, and dispatches row clicks via
  `onChannelClick` instead of the v0.3.x `onJoinChannel(name) → switchChannel`
  chain.

Plus, regardless of mode, the four Spec §4.4 sub-section headers are now
rendered: "Public listed" / "Public unlisted (accessible)" / "My private
channels" / "Archived". Each sub-section is alpha-sorted by name via
`localeCompare` (SORT-LOCK invariant). A non-`'alphabetical'` `sortKey`
prop logs a `console.warn` and the component still alpha-sorts.

The 4 new props are exactly the contract written into the Wave 0 stub
JSDoc block on `611b305`. Names not changed. Order in the destructure
groups the new props after the existing ones to keep the legacy
`store`/`onClose`/`onJoinChannel` positions stable.

## 4. Code changes (key excerpts)

### `web/src/components/ConversationBrowser.svelte` — 4 new props

```svelte
let {
  store,
  onClose,
  onJoinChannel,
  filterValue,
  sortKey,
  embedded = false,
  onChannelClick,
} = $props();
```

### Dual-mode filter input

```svelte
let internalFilter = $state('');

let effectiveFilter = $derived(
  typeof filterValue === 'string' ? filterValue : internalFilter,
);
```

`typeof filterValue === 'string'` (instead of truthy-check) so the
parent can pass empty string ("controlling, but no filter") without
the component falling back to internal $state.

### SORT-LOCK guard

```svelte
$effect(() => {
  if (typeof sortKey === 'string' && sortKey !== 'alphabetical') {
    console.warn(
      `[ConversationBrowser] sortKey="${sortKey}" ignored - SORT-LOCK invariant requires 'alphabetical'.`,
    );
  }
});
```

The actual row order is ALWAYS alphabetical because the `alphaSort()`
helper runs unconditionally in every `$derived` sub-section. The
warn exists so a future contributor wiring a different sort dropdown
upstream gets a loud signal that the prop is rejected.

### Sub-section partitioning (Spec §4.4)

```svelte
let publicListed = $derived(
  alphaSort(
    allChannels.filter(
      (c) => c?.mode === 'public' && c?.visibility === 'listed' && !c?.member && !c?.archived,
    ),
  ),
);
let publicUnlistedAccessible = $derived(
  alphaSort(
    allChannels.filter(
      (c) => c?.mode === 'public' && c?.visibility === 'unlisted' && c?.member === true && !c?.archived,
    ),
  ),
);
let archived = $derived(alphaSort(allChannels.filter((c) => c?.archived === true)));
let privateChannels = $derived(
  alphaSort(
    allChannels.filter(
      (c) => c?.mode === 'private' && c?.member === true && !c?.archived,
    ),
  ),
);
```

### Row click dispatch

```svelte
function handleRowClick(channel) {
  if (typeof onChannelClick === 'function') {
    onChannelClick(channel.id);    // embedded path → modal closes + switches
    return;
  }
  if (typeof onJoinChannel === 'function') {
    onJoinChannel(channel.name ?? channel.id);  // standalone path (v0.3.x)
  }
}
```

## 5. Behaviour matrix

| Mode | `filterValue` | `sortKey` | `onChannelClick` | Behaviour |
|---|---|---|---|---|
| standalone (default) | undefined | undefined | undefined | Internal filter input visible; row click calls `onJoinChannel(name)`. No warn. |
| standalone + custom sort | undefined | `'recency'` | undefined | console.warn fires; rows still alpha-sorted. |
| embedded | `'alpha'` | `'alphabetical'` | provided | Internal filter input HIDDEN; only rows matching `alpha` shown; row click calls `onChannelClick(id)`. No warn. |
| embedded | `''` | undefined | provided | Internal filter input HIDDEN; all rows shown; row click → `onChannelClick`. |
| embedded | undefined | undefined | provided | Embedded mode WITHOUT parent-supplied filter: falls back to the internal $state, but the internal input is still hidden (the parent should provide one externally if they want filtering UX). |

The last row is a defensive footgun mitigation; the modal in §2.13 always
provides `filterValue` in practice, but a hand-rolled embedded caller
won't crash.

## 10. Future work / follow-ups

* Step 2.13's integration of this component into `ChannelDirectoryModal`
  will exercise the embedded path end-to-end. If the modal's filter
  input debounces (which Phil's design spec doesn't require but is
  a reasonable UX), the component's `effectiveFilter` $derived
  picks up the debounced value automatically — no further work
  needed.
* The Spec-§4.4 sub-section headers use `<div class="browser-section-header">`
  rather than proper `<h3>` because the modal is the outer landmark
  (`role="dialog"` + `aria-label`) and the section headers are
  semantic-content-within-tab. A future a11y pass could promote
  these to `<h3>` with `aria-level` if Phil's design spec asks
  for it.
* The fetch-from-REST path in v0.3.x ConversationBrowser had its
  own loading + error UI. Removed in this refactor because the
  store already handles bootstrap loading + error states centrally
  (Step 2.6 / S-FIX). If a user opens the directory modal before
  the store finishes its first `/api/conversations` bootstrap, the
  modal will show "No channels yet" briefly, then auto-populate
  via the store's $derived reactivity. This is consistent with
  the rest of the app.
