## 1. Scope

Added the v0.4.0 `comms_conversation_delete` MCP tool plus the supporting
`ConversationMeta.mark_deleted` soft-delete method and `deleted_at` /
`deleted_by` fields.  The tool implements the two-phase contract from the
Design Spec §4.5: a `confirm=False` pre-flight that returns
`{"error": "confirm_required", "message_count": N, "member_count": N}` so
the web client can render a type-name confirmation modal, and a
`confirm=True` branch that executes the full 5-step soft-delete (retained
orphan banner on the per-conv `messages` topic, lifecycle event on the
global `system/conversations` topic, on-disk soft-delete via meta.json,
retained-clear of every member's per-conv presence row, and membership
drop in the registry).  Authorization is v0.4.0 **creator-only**
(`participant.name == meta.created_by`); the broader "creator OR active
human" rule is left for v0.4.1.

Nineteen new pytest specs cover every branch: pre-flight pre-flight no-op,
authorization rejection, full 5-step happy path with payload-level
assertions on every published topic, on-disk persistence of the
soft-delete fields, registry-side membership drop, reserved-name
rejection (`general` / `system`), unknown-conversation rejection,
double-delete rejection, unknown-key rejection, and three smoke specs in
`test_mcp_tools.py` against the shared `publish_spy` / `sample_participant`
fixtures.

## 4. Code changes (key excerpts)

### `conversation.py` — soft-delete fields + method

```python
class ConversationMeta(BaseModel):
    ...
    deleted_at: str | None = Field(default=None, description=...)
    deleted_by: str | None = Field(default=None, description=...)

    def mark_deleted(self, deleted_by: str) -> None:
        """Mark this conversation as soft-deleted (in-memory only)."""
        self.deleted_at = now_iso()
        self.deleted_by = deleted_by

    @property
    def is_deleted(self) -> bool:
        return self.deleted_at is not None and self.deleted_by is not None
```

### `mcp_tools.py` — 5-step soft-delete (confirm=True branch)

```python
# Step 1: final RETAINED orphan-banner system message
orphan_payload = {
    "type": "deleted",
    "conversationId": conversation,
    "deletedBy": participant.name,
    "timestamp": ts,
}
await publish_fn(messages_topic, json.dumps(orphan_payload).encode(), retain=True)

# Step 2: global lifecycle event (publish_conversation_event wire-format)
lifecycle_payload = {
    "type": "conversation_deleted",
    "name": conversation,
    "deleted_by": participant.name,
    "ts": ts,
}
await publish_fn("claude-comms/system/conversations", ...)

# Step 3: persist soft-delete to meta.json
meta.mark_deleted(participant.name)
save_meta(meta, conv_data_dir)

# Step 4: retained-clear every member's presence row
for member in members:
    await publish_fn(f"claude-comms/conv/{conversation}/presence/{member.key}",
                     b"", retain=True)

# Drop registry-side memberships so sidebars built from the registry
# (not meta.json) immediately reflect the deletion.
for member in members:
    registry.leave(member.key, conversation)

return {"deleted": True, "conversation_id": conversation}
```

### `mcp_server.py` — new MCP tool registration

```python
@mcp.tool()
async def comms_conversation_delete(
    key: Annotated[str, Field(description="Your participant key")],
    conversation: Annotated[str, Field(description="Conversation to delete")],
    confirm: Annotated[bool, Field(description=...)] = False,
) -> dict[str, Any]:
    """Soft-delete a conversation. Creator-only. Two-phase: ..."""
    _touch(key)
    assert _publish_fn is not None, "Publish function not initialised"
    return await tool_comms_conversation_delete(
        _get_registry(), _get_store(), _publish_fn,
        key=key, conversation=conversation, confirm=confirm,
        conv_data_dir=_get_conv_data_dir(),
    )
```

## 7. Findings surfaced during work

- **(S) `publish_conversation_event` helper not reused.**  The brief
  notes "Reuse that helper if possible; if not, add a new event-type
  constant."  The helper lives in `mcp_server.py`, which imports from
  `mcp_tools.py`.  Calling it from `tool_comms_conversation_delete`
  would invert that direction and create a circular import on first
  load.  Mirrored its wire-format inline instead (no new event-type
  constant — `"conversation_deleted"` is already documented in the
  helper's docstring as a planned v0.4.0 type).  If the helper is ever
  moved out of `mcp_server.py` (e.g. into a `lifecycle_events.py`), the
  inline copy here becomes a 3-line `await publish_conversation_event(...)`
  call.
- **(S) Authorization comparison is by name string, not by key.**
  `ConversationMeta.created_by` stores the participant's display name at
  creation time (per `tool_comms_conversation_create` line 1838:
  `created_by=participant.name`).  This means a participant who renames
  themselves between creating a conversation and trying to delete it
  loses the ability to delete it.  Two options for v0.4.1: (a) store
  the creator's 8-hex key instead, with a one-time migration to look up
  the key by name on existing meta files; or (b) accept that rename
  forfeits delete rights and surface this in the UI.  I went with the
  existing name-based field for v0.4.0 to avoid a meta.json schema
  change in a step focused on the new tool.  Flagged for v0.4.1.
- **(S) Empty-payload retained-clear assumes broker support.**  Mosquitto
  / aiomqtt honor the MQTT 5 spec §3.3.1 "zero-byte payload + retain=True
  drops the retained message" contract.  amqtt (our daemon broker) also
  honors this — verified against `src/claude_comms/broker.py` retained-
  message handling in the v0.2.0 work.  No regression risk for our
  default stack; non-default external brokers may need a config check.
- **(S) Soft-delete leaves `claude-comms/conv/{id}/messages` history on
  disk and in the in-memory `MessageStore`.**  Per the brief: "Soft-delete:
  keep row with `deleted_at` timestamp; don't drop history hard until a
  future purge job."  Live subscribers will see the orphan banner; new
  subscribers will see whatever the broker still retains plus the orphan
  banner.  A future purge job (v0.4.x) should: (a) call `MessageStore.clear(conv)`,
  (b) `os.unlink` the JSONL log, (c) iterate retained messages on
  `claude-comms/conv/{id}/+` and clear them.  Out of scope here.
- **(I) Tool count delta**: registered tool count went from 22 to 23 (the
  new `comms_conversation_delete`).  Verified via the inline sanity
  check in §5.

## 9. Outstanding concerns

**1. Key-based creator identity is a v0.4.1 follow-up.**  See §7
finding 2.  Track in `plans/v0.4.0-release.md` §III "Deferred" or open
a v0.4.1 issue.

**2. Retained-history purge job is a v0.4.x follow-up.**  See §7 finding
4.  Once Wave A integrates, a hard-delete job that runs on a schedule
(or on demand from an admin tool) should clear the per-conv JSONL log
and any retained chat-history messages.  Not blocking for v0.4.0
because soft-deleted conversations are correctly invisible to clients
that go through the sidebar (which subscribes to
`claude-comms/system/conversations` and now receives
`conversation_deleted` events).

**3. End-to-end smoke (real broker) not attempted by this agent.**  The
unit specs cover the wiring; an orchestrator-level smoke (start daemon,
create + delete a conversation via the web UI, watch the MQTT topics
with `mosquitto_sub`) is left for the integration step after Wave A
merges.

**4. Sidebars built from `ParticipantRegistry.conversations_for()` need
no extra work.**  Step 4 calls `registry.leave(member.key, conv)` for
every member, so the registry view is consistent with meta.json
post-delete.  Sidebars built from `list_all_conversations(conv_data_dir)`
will see the new `deleted_at` field and should filter out soft-deleted
rows — this filter belongs in the Step 2.3 / web-UI work and is out of
scope here.

**5. CLI parity.**  The brief is scoped to MCP only; no CLI command
(`claude-comms conversation delete <name>`) is added.  Could be a
v0.4.x ergonomic add — mention in the v0.4.1 polish brief.
