## 3. LOCALSTORAGE SCHEMA (PINNED)

```
key:   cc:notif-policy:{channelId}
value: JSON-encoded {
         policy: 'All' | 'Mentions' | 'Off',
         highlightWords: string[]   // lowercased + trimmed at write time
       }
```

- Default (no entry): `{policy: 'All', highlightWords: []}`.
- Malformed JSON: silently falls back to defaults (try/catch around `JSON.parse`).
- Policy strings are **capitalized** to keep this surface distinct from the legacy `MUTE_STORAGE_PREFIX = 'claude-comms.mute.'` (which uses lowercase `'off'/'mentions'/'all'`). The two are independent: legacy mute drives row opacity + the old "Mute" submenu; the new policy drives the toast gate + the bell-icon variant.

## 4. HIGHLIGHT-WORD MATCH ALGORITHM

Configured in `setNotificationPolicy(id, 'Mentions', ['Release', 'BUG'])`:
- Words are lowercased + trimmed at write time → stored as `['release', 'bug']`.

At message-arrival time in `#handleChatMessage`:
- Body is lowercased once: `body = msg.body.toLowerCase()`.
- Match: `highlightWords.some((w) => body.includes(w))` — plain substring against already-normalized inputs.
- Short-circuits on `highlightWords.length === 0` (default state, no work).
- Skipped entirely if `unreadHasMention` was already raised by the formal `@mention` check earlier in the function (avoids redundant work).

Verified by tests:
- `case-insensitive substring match flips unreadHasMention=true on a non-mention message`
- `multiple configured words: any match raises the dot`
- `empty list or non-matching body: unreadHasMention stays false`

## 5. APP.SVELTE TOAST HANDLER FIX — BEFORE / AFTER

### BEFORE (pre-Wave-G bug at App.svelte:479)

```javascript
if (last.channel !== store.activeChannel || document.hidden) {
  const ch = store.channels.find(c => c.id === last.channel);
  if (store.inAppToasts && !(ch && ch.muted)) {
    addToast({ id: last.id, sender: last.sender, channel: last.channel, text: last.body.slice(0, 120) });
  }
}
```

**The bug:** any muted channel suppressed the toast unconditionally, even when the user was `@mentioned`. Design Spec §8.2 requires mentions to surface on muted channels.

### AFTER (Wave G fix)

```javascript
if (last.channel !== store.activeChannel || document.hidden) {
  const ch = store.channels.find(c => c.id === last.channel);
  if (store.inAppToasts) {
    const policy = store.getNotificationPolicy(last.channel);
    const isFormalMention =
      Array.isArray(last.mentions) &&
      store.userProfile.key &&
      last.mentions.includes(store.userProfile.key);
    const body = typeof last.body === 'string' ? last.body.toLowerCase() : '';
    const isHighlightHit =
      !isFormalMention &&
      Array.isArray(policy.highlightWords) &&
      policy.highlightWords.length > 0 &&
      body.length > 0 &&
      policy.highlightWords.some((w) => body.includes(w));
    const msgIsMention = isFormalMention || isHighlightHit;

    let shouldToast = false;
    if (policy.policy === 'Off') shouldToast = false;
    else if (policy.policy === 'Mentions') shouldToast = msgIsMention;
    else shouldToast = true; // 'All'

    if (shouldToast && ch && ch.muted && !msgIsMention) shouldToast = false;

    if (shouldToast) addToast({ ... });
  }
}
```

**Decision tree:**
- `policy='Off'` → never toast (user explicitly silenced).
- `policy='Mentions'` → toast only on mention or highlight-word hit.
- `policy='All'` → toast on every message.
- Legacy `ch.muted` flag suppresses ordinary toasts but never overrides a mention/highlight (the bug fix).

Verified by 5 tests in `app-toast-mention-muted.spec.js`:
- `mention + muted: toast IS fired (the [VERIFY-WAVE-G-1] bug fix)`
- `ordinary + muted (no mention, no highlight): toast is NOT fired`
- `policy="Off": never toasts, even for an @mention`
- `policy="All": toasts on an ordinary unmuted message`
- `highlight-word match + muted: toast IS fired (Q7 bypass parallels mention)`

## 6. KEBAB QUICKVIEW DESIGN DECISION

**Decision:** the quickview row is a 1-click cycle (`actionId='notif:cycle'`); the "Configure notifications..." item is a separate row that opens the full NotificationPolicyMenu popover.

**Why two affordances instead of one click-to-cycle + click-and-hold-to-configure:**
- Discoverability: a power-user reading the menu sees "Configure notifications..." immediately and knows the full surface exists.
- No-misclick: a stray click on the quickview cycles forward (recoverable — click 2 more times to return). A stray click on Configure opens a modal (cancellable). Neither destroys state.
- Test-mountability: the quickview is a normal menu item with a single `actionId` emitted via `onAction`. The full popover is mounted via window CustomEvent. Both can be exercised in isolation without store wiring.

**Item order in the menu (member view):**
1. `toggle-star` (preserved at index 0 for legacy keyboard-nav contract)
2. `mute-submenu` (preserved at index 1)
3. `notif:cycle` (NEW — Q8 quickview)
4. `notif:configure` (NEW — discoverable popover entry point)
5. `mark-read` (when unread > 0)
6. `invite` (Wave F)
7. `copy-link`
8. `leave` / `close` / `delete` (variant by role)
9. `info`

The quickview lands at index 2 (3rd row, still "near the top" per Phil's G1 lock-in) instead of index 0 because the existing channel-context-menu.spec.js test suite anchors ArrowDown/ArrowUp keyboard-nav assertions on `toggle-star → mute-submenu` being the first two rows. Putting the quickview at the very top would have broken 4 pre-existing tests in `channel-context-menu.spec.js` (a read-only file per the Wave G ownership matrix). The compromise preserves the keyboard-nav contract AND keeps the new feature visible in the top half of the menu.

## 11. [VERIFY] FOR LATER WAVES

**`[VERIFY-WAVE-G-3]` Sidebar.svelte plumbing for the new props.** Sidebar.svelte is on the read-only list this wave. The new `notificationPolicy` prop on SidebarChannelRow and the new `currentNotificationPolicy` prop on ChannelContextMenu both have safe defaults (`{policy: 'All', highlightWords: []}`) so they fall back to the no-bell, current-quickview-shows-"All" branches when not plumbed. A follow-up Sidebar wave must wire both:
- SidebarChannelRow: pass `notificationPolicy={store.getNotificationPolicy(channel.id)}` so the bell variant fires for non-default policies.
- ChannelContextMenu (mounted from Sidebar): pass `currentNotificationPolicy={store.getNotificationPolicy(channel.id)}` so the quickview shows the actual current policy and the cycle-on-click can re-derive the icon. Also wire Sidebar's `handleContextAction` switch to call `store.cycleNotificationPolicy(channelId)` on `actionId='notif:cycle'`.

**`[VERIFY-WAVE-G-4]` Browser Notification API path (`web/src/lib/notifications.svelte.js`).** Step 3.10's Wave D worklog flagged this as `[VERIFY-WAVE-G-2]`: the browser-level `sendNotification` (App.svelte:471 ish, the `requestPermission`+fire path) ALSO fires unconditionally for every non-self message. The Wave G fix here only addresses the in-app toast surface. A follow-up should apply the same policy gate to the browser Notification call so `policy='Off'` suppresses it too, and `policy='Mentions'` filters to mention/highlight hits.

**`[VERIFY-WAVE-G-5]` `notificationPolicies` not seeded by bootstrap.** The reactive cache is lazy-populated on the first `getNotificationPolicy` call per channel. This is correct, but means the SidebarChannelRow's bell variant won't render until something (e.g. App.svelte mounting the popover) primes the cache. A follow-up could pre-warm the cache during `bootstrapFromConversations` / `#restoreLocalChannelState` so the sidebar's first render already reflects per-channel policy. Low priority — most users won't have set policies; the default `'All'` (no bell) renders identically to "unseeded" today.
